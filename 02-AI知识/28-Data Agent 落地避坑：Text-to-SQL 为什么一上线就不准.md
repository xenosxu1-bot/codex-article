> 一句话结论：Text-to-SQL 能做演示，但企业级 Data Agent 需要语义层、权限、指标治理和多阶段校验。

帮我查一下这个月华东区的新客转化率。这句话看似适合 Text-to-SQL，真正上线后却很容易翻车。

原因是企业问题通常不只是把自然语言翻译成 SQL。系统要知道华东区的组织边界、新客的定义、转化率的口径、哪些表可信、哪些字段有延迟、用户是否有权限查看。这些上下文缺一块，SQL 可能语法正确，业务结论却是错的。

InfoQ 关于企业级数据 Agent 的文章把落地路径拆成多个阶段：从取数，到智能分析，再到工程化治理。它提醒我们，Data Agent 的难点不是会不会写 SQL，而是能不能在真实业务语义里稳定工作。

## 为什么这个话题现在值得写

第一，工具能力正在从回答走向执行。当 AI 可以浏览网页、读文件、调用工具、运行命令或操作桌面，用户面对的就不再是一个聊天框，而是一个需要被安排、约束和验收的任务系统。

第二，企业和个人都开始遇到同一个问题：AI 输出越来越快，但可交付结果并不会自动变多。真正影响效率的，是任务定义、上下文质量、权限边界、验证方式和复盘机制。

第三，这个方向有连续写作价值。它既能写产品趋势，也能写工作方法；既能写企业落地，也能写普通人的使用教程。

## 读者最容易误解的地方

很多人会把这个话题理解成换一个更强模型。这会漏掉关键点：模型只是能力来源，工作流才决定能力能不能变成结果。

另一种误解是把 Agent 当成全自动员工。更稳的做法，是把它看作能承担部分执行动作的协作者。人仍然负责目标、边界、判断和最终责任。

## 可以直接采用的做法

- 给指标建语义卡：定义、计算公式、负责人、更新时间。
- 把 SQL 生成和业务解释分开验收。
- 对高风险查询设置只读、限表、限行和审批。
- 让 Agent 输出 SQL、口径解释和不确定项。
- 把常见问题沉淀为可复用 Skill，而不是每次临场猜。

## 一个低风险试点

选一个每周都会重复、但失败成本不高的任务，例如资料整理、会议纪要、竞品摘要、代码小修复、表格清洗或文章初稿。

把任务拆成四段：

1. 输入：给 AI 哪些资料，哪些资料不能用；
2. 执行：让 AI 只做一个明确阶段；
3. 校验：要求列出来源、假设和待确认项；
4. 沉淀：把好用提示词、文件结构和检查表保存下来。

跑通一次后，再决定是否扩大权限和范围。

## 写作延展方向

这篇可以继续拆成三个角度：

- 面向普通读者：如何把 AI 用成稳定工作流；
- 面向团队管理者：如何给 Agent 设置权限和验收机制；
- 面向开发者：如何把 Agent 接入工具、数据和自动化流程。
