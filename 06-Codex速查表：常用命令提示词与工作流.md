# Codex 速查表：常用命令、提示词与工作流
副标题：收藏这一篇，遇到任务直接套模板。

![Codex 速查表封面](images/06-cheatsheet-cover-cn.jpg)

## 开篇
Codex 好不好用，不看提示词有多长，关键看场景、上下文和验收标准有没有说清楚。
这是一份 Codex 快速上手速查表：入口怎么选、常用命令怎么用、不同任务怎么开口、最后怎么验收。

## 一、先选入口

| 入口 | 适合场景 | 典型用法 |
| --- | --- | --- |
| Codex App | 新手、可视化开发、看 diff、跑前端预览 | 修 bug、做功能、审查改动 |
| Codex CLI | 终端用户、快速任务、脚本化 | `codex "fix failing tests"` |
| IDE Extension | 边看代码边问、选中文件解释 | 解释当前文件、按选中代码修改 |
| Cloud/GitHub | 远程任务、Issue、PR 辅助 | 让 Codex 处理独立任务 |

可以这样选：

- 想看过程：用 App。
- 想快：用 CLI。
- 想贴着代码问：用 IDE。
- 想远程交给它跑：用 Cloud/GitHub。

## 二、常用命令速查

![Codex 常用命令速查图](images/10-cheatsheet-cn.jpg)
常见命令：

```text
/plan      先让 Codex 规划，不急着改文件
/review    审查当前 diff，找 bug、回归、安全风险
/model     切换模型或推理强度
/status    查看当前模型、权限、上下文
/diff      查看当前 Git 改动
```

常见 CLI 用法：

```bash
codex
codex "explain this repository"
codex "fix the failing test and run the smallest relevant verification"
codex exec "review the current diff for correctness and missing tests"
```

## 三、通用提示词结构
把任务写成这四段，Codex 成功率会明显提高：

```text
目标：
[你希望 Codex 完成什么]

上下文：
[相关文件、报错、截图、复现步骤、参考实现]

约束：
[不要改什么、不要新增依赖、保持 API、范围限制]

完成标准：
[跑什么命令、输出什么结果、怎样算完成]
```

示例：

```text
目标：
修复登录页在移动端按钮文字溢出的问题。

上下文：
- 相关文件：src/pages/login.tsx、src/components/Button.tsx
- 截图显示按钮文字超出容器。

约束：
- 不重构按钮组件。
- 不修改登录接口逻辑。
- 优先复用现有 CSS token。

完成标准：
- 修复桌面和移动端布局。
- 启动本地页面并截图验证。
- 最后说明改了哪些文件和验证结果。
```

## 四、常见任务模板

### 1. 解释项目

```text
请先只读分析这个项目，不要修改文件。

请输出：
1. 项目技术栈。
2. 主要目录作用。
3. 启动、构建、测试命令。
4. 新人最该先读的 5 个文件。
5. 你发现的风险或不确定点。
```

### 2. 修复 bug

```text
请按 Debug 闭环修复这个 bug：

问题：[现象/报错/复现步骤]

要求：
- 先复现或写最小失败测试。
- 定位根因后再修改。
- 只做最小修复。
- 跑最小相关测试。
- 最后汇报根因、修改、验证和风险。
```

### 3. 新增功能

```text
请实现 [功能名称]。

上下文：
- 参考现有实现：[文件]
- UI/接口要求：[说明]
- 不要改动：[约束]

完成标准：
- 功能可用。
- 补充必要测试。
- 跑 lint/typecheck/相关测试。
- 最后列出用户如何验证。
```

### 4. 代码审查

```text
请以 code review 方式审查当前 diff。

优先级：
1. 正确性 bug。
2. 安全风险。
3. 行为回归。
4. 缺失测试。
5. 可维护性问题。

请给出文件和具体位置，不要只给泛泛建议。
```

### 5. 重构

```text
请先制定重构计划，不要修改文件。

目标：[要改善什么]
约束：
- 保持外部行为不变。
- 分阶段提交，每阶段可验证。
- 不混入功能变化。

请输出：
1. 当前问题。
2. 推荐重构路径。
3. 每一步的风险和验证方式。
```

## 五、让 Codex 更稳定的 8 条习惯

1. 给文件路径，比让它全局猜更好。
2. 给报错全文，比只说“挂了”更好。
3. 给完成标准，比只说“搞定”更好。
4. 大任务先 `/plan`，小任务直接做。
5. 高风险任务先只读调研。
6. 修 bug 要求先复现。
7. 提交前让 Codex `/review` 当前 diff。
8. 重复出现的规则写进 `AGENTS.md`。

## 六、最后汇报模板
建议每次让 Codex 最后这样汇报：

```text
请用这个格式结束：

修改内容：
- [文件]：[做了什么]

验证结果：
- [命令]：[通过/失败/未运行原因]

风险与后续：
- [剩余风险或建议]
```

这能帮助你快速判断：它到底做了什么，是否真的验证过，还剩什么风险。

## 结尾
Codex 最好用的方式，不是把它当成万能聊天框，而是把它放进工程流程里：先给上下文，再给边界，最后要证据。
把这份速查表收藏起来，遇到任务直接套模板，你会明显感觉 Codex 从“有时聪明”变成“稳定可用”。
