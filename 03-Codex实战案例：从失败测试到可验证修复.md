# Codex 实战案例：从失败测试到可验证修复

副标题：不要只让 Codex “帮我修一下”，而是让它复现、定位、最小修改、跑验证。

![Codex 实战案例封面](images/03-case-cover-cn.jpg)

## 开篇

很多人第一次用 Codex 修 bug，常见提示词是：

```text
这个测试挂了，帮我修一下。
```

这句话当然能用，但它把关键判断都交给了模型：该不该先复现？该改业务逻辑还是测试？要不要跑全量测试？修完怎么证明没引入回归？

真正稳定的用法，是把 Codex 当成一位会进项目工作的工程师，让它按“Debug 闭环”执行：先复现，再定位，再最小修改，最后验证并汇报。

## 一、给 Codex 一个可以执行的任务，而不是一个情绪化问题

不推荐：

```text
页面有问题，帮我看看。
```

推荐：

```text
请修复订单列表筛选失效的问题。

上下文：
- 复现路径：进入 /orders，选择状态 Paid，列表仍显示全部订单。
- 相关文件可能在 src/pages/orders 和 src/api/orders。
- 不要重构筛选组件，只修复本次 bug。

完成标准：
- 先复现并说明根因。
- 做最小改动。
- 补充或更新相关测试。
- 跑最小相关测试，并在最后汇报命令和结果。
```

这个提示词的重点不是“更长”，而是有四个要素：

- 目标：修复什么问题。
- 上下文：从哪里开始查。
- 约束：不要做什么。
- 完成标准：怎样算完成。

## 二、第一步：要求 Codex 先复现问题

如果 Codex 没有先复现，就直接改代码，风险会变高。你可以明确要求：

```text
先不要修改文件。请先阅读相关代码，找出可能复现路径，并告诉我你准备用哪个测试或命令复现问题。
```

适合让 Codex 做的复现动作：

- 运行失败测试。
- 根据错误日志定位入口。
- 用最小命令重现问题。
- 前端项目用浏览器或截图确认页面状态。
- 如果没有测试，先写一个能暴露问题的最小测试。

这里有一个小技巧：让 Codex 在动手前先说“预期失败点”。例如：

```text
在修改前，请先说明你认为当前失败的原因，并用一个最小测试证明它确实失败。
```

这样可以减少“修了另一个问题”的概率。

## 三、第二步：定位根因，而不是追着报错改

优秀的 Debug 不是看到哪里红就改哪里，而是回答三个问题：

1. 错误在哪里出现。
2. 数据从哪里变错。
3. 为什么之前没有被测试覆盖。

你可以这样要求 Codex：

```text
请沿着数据流排查：从 UI 参数、请求构造、接口返回、状态更新到渲染输出。只在确认根因后再修改。
```

如果是后端接口问题，可以换成：

```text
请沿着请求链路排查：路由、参数校验、服务层、数据库查询、返回结构。确认根因后再给出最小修复。
```

如果 Codex 的结论不够具体，继续追问：

```text
请指出具体文件、函数和条件分支。不要只说“筛选逻辑有问题”。
```

## 四、第三步：坚持最小改动

Codex 很适合做重构，但修 bug 时，过度重构会让 review 变困难。

建议在提示词里写清楚：

```text
本次只修复 bug，不做无关重构，不调整公开 API，不新增依赖。除非必须，请保持 diff 尽量小。
```

修复过程中你可以观察 Codex 的行为：

- 它是否只改了相关文件。
- 它是否顺手格式化了大文件。
- 它是否把测试一起改到“永远通过”。
- 它是否修改了对外接口或配置。

如果发现 diff 变大，立刻让它收敛：

```text
当前 diff 超出了本次 bug 范围。请解释每个文件为什么必须修改，并撤回不必要改动。
```

## 五、第四步：让 Codex 自证正确

修完后，不要只看 Codex 说“已完成”。真正有价值的是验证证据。

你可以要求最后输出：

```text
请按下面格式汇报：

1. 根因：一句话说明。
2. 修改：列出改了哪些文件和为什么。
3. 验证：列出运行过的命令、结果和未运行的原因。
4. 风险：还可能有哪些边界情况。
```

一个好的最终汇报应该长这样：

```text
根因：订单状态筛选参数被 UI 传成 paid，但 API 查询只接受 PAID。
修改：在请求构造层统一转换枚举值，并补充状态筛选测试。
验证：pnpm test orders-filter 通过；pnpm typecheck 通过。
风险：如果后续新增状态，需要同步更新枚举映射。
```

## 六、实战提示词模板

你可以直接收藏这段：

```text
请修复下面的问题，并按 Debug 闭环执行。

问题：
[写清楚现象、报错、复现路径]

上下文：
- 相关文件可能是：[文件或目录]
- 相关命令：[测试/构建/启动命令]
- 参考行为：[正确行为或旧版本行为]

约束：
- 先复现或写最小失败测试，再修改。
- 只做最小修复，不做无关重构。
- 不新增依赖，除非先说明原因并获得确认。

完成标准：
- 说明根因。
- 列出修改文件和原因。
- 跑最小相关测试。
- 如果无法验证，请说明阻塞原因和建议验证方式。
```

## 七、什么时候该用 Plan mode 或 Review

任务越复杂，越不要急着让 Codex 动手。

适合先用 Plan mode 的场景：

- 涉及多个模块。
- 不确定改动边界。
- 需要迁移数据结构。
- 需要同时改前端、后端和测试。

适合让 Codex 做 Review 的场景：

- 你或 Codex 已经改完代码。
- diff 比预期大。
- 涉及权限、支付、登录、数据删除。
- 要提交到远程仓库前。

一句话：Plan mode 用来防跑偏，Review 用来防漏坑。

## 结尾

Codex 修 bug 的核心不是“让它一次猜对”，而是把工程闭环交给它执行：复现、定位、最小修复、验证、汇报。

当你把任务写到这个粒度，Codex 就不再只是回答问题，而是在帮你按专业开发流程把问题关掉。
