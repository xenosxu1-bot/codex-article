# AGENTS.md

## 教程写作规范

- 标题要简洁，优先让读者一眼看懂主题。
- 正文少 AI 味，避免空话、套话、翻译腔和过度解释。
- 使用中文文章标点，中文段落中不要混用英文逗号、句号。
- 段落保持紧凑，普通段落之间尽量少留空行；标题、图片、表格、列表和代码块前后保留必要空行。
- 文章以实用为主，减少重复介绍；同一概念已经在其他文档讲过时，只保留必要引用和核心结论。
- 删除填充短语，少用“可以理解为”“简单说”“需要注意的是”等解释腔。
- 避免金句式结尾，结尾优先给具体动作或使用场景。
- 少用三段式排比，能用两项或四项就不要强行凑三项。
- 长句优先拆短，具体信息优先于抽象判断。
- 图片较多时，减少重复表格和重复说明。
- 正文不使用中文全角引号；需要突出时使用 Markdown 加粗。
- 正文不使用 Markdown 分割线；不设置发布前检查、参考资料、参考文献、资料来源或延伸阅读等区块。真实性核验在写作过程中完成，正文只保留经核验的结论与必要边界。
- 标题默认不使用问号；只有用户明确要求保留时才使用。

## 图片规范

- 配图优先使用官方图标、官方界面图或用户提供的真实素材。
- 没有官方素材时，再做自定义设计图。
- 自定义图要保持中文可读、颜色对比清晰、布局不遮挡文字。
- 文章封面默认使用 2.35:1 横版，推荐 2350 × 1000 px；左侧主副标题、右侧主题视觉主体，深色科技基底，配色必须随文章主题调整。
- 封面先生成原创视觉底图，再用确定性排版叠加并校对中文主副标题；禁止使用错字、乱码、伪造品牌 Logo、伪造官方界面或与文章无关的通用画面。
- 封面提示词与动态配色规则以 `09-工具脚本/封面提示词模板-V1.md` 为准；每累计 10 张封面后，根据反馈和表现复盘优化版本。
- 后续新增或重写文章统一按 `07-资料与流程/后续文章生成方案.md` 执行：先 Brief，再正文，再外置登记元信息，再配图，再入库索引，再发布记录；如果用户只要草稿，明确不入库不提交。
- 提交前检查所有 Markdown 图片引用，确保文件存在且显示位置合理。
- 后续新增、整合或重写文章，正文必须至少配置 1 个阅读缓冲模块：优先使用本地正文插图；如果暂不配图，必须改用“视觉速读 / 方法卡 / 对比卡 / 检查清单 / 流程卡”等可扫读版式，避免整篇都是连续纯文本。
- 正文插图不得放在首行，也不得复用封面图；推荐放在一句话结论之后、第一组正文小标题之前，用于帮助读者先抓住文章结构。

## 开源项目文章生产流程

- 写开源项目公众号文章时，默认先核对 GitHub README、官方文档和核心安装/安全页面，资料时间写入内部核验记录。
- 正文至少覆盖项目定位、新手快速上手、核心功能、可复现使用案例、适用边界和新手避坑。
- 配图默认包含 1 张封面和至少 2 张正文信息图；正文图要解释架构、流程或案例，不复用封面。
- 使用案例必须标注为可复现模板或已验证经历，不虚构客户案例、效率数据或生产经验。

## 提交前检查

- 检查 Markdown 链接、图片引用和代码块是否完整。
- 检查是否存在乱码、明显错别字和过时文件名。
- README 有新增文章或素材时，需要同步更新目录和图片说明。

## 选题阶段追踪

- 新增或重写文章前，先登记或更新 `07-资料与流程/选题库.md`，并对照 `07-资料与流程/已发布文章去重登记.md` 做查重；不得只换标题重复旧文。
- 每启动一篇后续文章，先把对应选题从 `S0 灵感池` 更新到 `S1 已立题`；后续按 `S2 已做资料/Brief`、`S3 初稿中`、`S4 待质检`、`S5 待发布`、`S6 已入库`、`S7 已复盘` 逐步记录，不跳过状态登记。
- 如果选题暂缓、合并或放弃，分别标记为 `HOLD` 或 `DROP`，并写清原因、恢复条件或合并去向。
- 如果选题与已发布文章高度重叠，必须在选题库写清新增角度：新读者、新场景、新案例、新模板、新资料或团队版；否则不进入正文写作。
- 进入正文前必须先形成 Brief，至少包含目标读者、核心问题、不写什么、核心观点、交付物、资料要求、风险点和关联旧文。
- 正式入库或推送后，同步更新 `选题库.md`、`已发布文章去重登记.md`、状态跟踪文件、总索引、分类索引、系列索引、标签索引和发布记录，确保主题、阶段、编号、路径和 GitHub 状态可追溯。

## 选题库与同步管控

- 选题库、已发布去重登记和同步管控机制分别以 `07-资料与流程/选题库.md`、`07-资料与流程/已发布文章去重登记.md`、`07-资料与流程/选题同步与规划管控机制.md` 为准。
- `main` 作为个人生产与规划工作区，`prod` 作为确认后的共享知识库展示区；后续项目修改默认先落 `main`，用户确认后再同步 `prod`。
- 同一 GitHub 仓库的不同分支不是隐私边界；真正私密内容不得放入公开仓库或公开分支历史。
- 每次新增选题、发布文章或同步分支后，都要能从选题库、资产登记表、发布记录和 Git 状态追溯来源与去向。

## 文章编号与上传

- 后续新增或重写文章统一按 `07-资料与流程/后续文章生成方案.md` 执行：先 Brief，再正文，再配图，再入库索引，再发布记录；如果用户只要草稿，明确不入库不提交。
- 新文章使用 `NN-中文标题.md` 命名，但不要再放在仓库根目录；必须放入对应中文分类目录（如 `01-工具教程/`、`02-AI知识/`、`03-好文方法/`）。发布前检查 `00-知识库导航/知识库总索引.md`，使用下一个未占用的两位数编号；不得重编号已发布文章。
- 新增配图必须使用“文章编号-中文文章标题-用途”命名，且标题与 `07-资料与流程/文章资产登记表.md` 中的正式标题一致：封面为 `NN-中文文章标题-封面.扩展名`，正文插图为 `NN-中文文章标题-正文插图01.扩展名`；封面放入 `08-素材库/图片/文章封面/`，正文插图放入 `08-素材库/图片/正文插图/`。重命名时必须同步更新全部本地引用、README 与知识库索引。
- 新增文章时同步更新 `README.md`、`00-知识库导航/知识库总索引.md`、`分类索引.md`、`系列索引.md`、`标签索引.md` 和 `07-资料与流程/发布记录.md`；读者可见的 README、首页和索引使用连续阅读序号，文件名、图片名、资产登记表和发布记录保留历史文章编号，避免链接和记录失真。
- 更新 README、首页或知识库索引后，必须运行 `09-工具脚本/重建知识库索引.py` 或按其输出规则重建，确认读者可见序号连续、链接可访问、资产登记表仍保留历史编号。
- 除非任务明确要求“仅草稿”或“不上传”，文章和素材完成本地校验后提交并推送 `main`；推送前先获取远端更新，禁止强推。


## main 到 prod 晋级发布规则

- 默认所有项目修改先在 `main` 完成：本地校验通过后，提交并推送 `main`，不得直接把未确认改动提交到 `prod`。
- `prod` 只作为确认后的稳定发布分支；只有用户明确确认“同步到 prod”“发布到 prod”或同等表达后，才把 `main` 中已确认的提交合并或同步到 `prod`。
- 同步 `prod` 前先确认 `main` 与 `prod` 的差异，优先保留 `prod` 已有的知识库导航、发布记录等稳定规则，再合入 `main` 的最新内容。
- 同步 `prod` 后必须至少完成文章质量扫描、Markdown 链接/图片检查和 Git 格式检查；检查通过后再推送 `prod`。
- 同步完成后向用户说明：`main` 提交、`prod` 提交、检查结果、是否已推送，以及当前工作区是否仍有未提交文件。

## 知识库目录约定

- 顶层只保留 GitHub/Codex 必需入口文件（`README.md`、`AGENTS.md`）和中文一级目录。
- 文章按主分类进入中文目录；交叉分类通过索引和标签解决，不复制正文文件。
- 素材、脚本、流程文档分别放入 `08-素材库/`、`09-工具脚本/`、`07-资料与流程/`；不保留整包发布归档，只保留正式文章实际引用的图片素材。

## 文章质量与微信发布兼容规则

- 正式文章正文只保留可发布内容，不展示元信息表、frontmatter 或隐藏元数据块。
- 正文不重复文章标题，不使用 H1；开头默认使用 `> 一句话结论：……`。
- 封面图外置到素材库和文章资产登记表，不作为正文第一张图重复展示。
- 每篇正式文章至少保留 1 个图文或视觉化版式模块；优先使用 `../08-素材库/图片/正文插图/编号-文章标题-正文插图01.png`，没有图片时必须用卡片、清单、对比块或流程块承接。
- 加粗必须克制：每篇建议不超过 6 处，超过 10 处发布前必须修复；禁止 `****`、相邻加粗、整句加粗和单字加粗。
- 微信公众号发布版原则上不保留围栏代码块；命令、提示词、模板和示例改成引用示例卡、编号清单或静态图片。Mermaid 必须转为静态图片后引用；确需代码教程时，发布前单独确认转换器支持。
- 正文不得保留 `[^1]` 这类脚注标记；需要来源时，改为正文内有边界表达，并把来源、访问日期、版权核验写入资产登记表或发布流程。
- 宽表格优先改成卡片、小节或编号清单；4 列及以上表格默认视为移动端 P1 问题。
- 不把“发布前请复核”“审核记录”“图片来源/参考资料”等内部流程区块放进公开正文；公开稿只可保留统一收束句：`本文发布前已完成事实、版权与格式检查。`
- 对没有点名来源的产品趋势，不写“当前一些产品……”这类过实判断；改成“许多 AI 写作/项目工具正在……”等类别表达。
- 结尾使用自然收束或下一步行动建议，不在知识库正文中写“点在看”“关注公众号”等平台 CTA。
- 发布前参考 `07-资料与流程/文章质量检查与微信发布规范.md`，并运行 `09-工具脚本/文章质量扫描.py` 做 P0/P1 检查。

## 后续任务自动提交约定

- 用户已确认：后续本知识库任务默认在本地校验通过后提交并推送到 `origin/main`，除非用户明确要求“仅本地修改”“仅草稿”“不要提交”或“不要推送”。
- 自动提交前必须先获取远端 `main` 最新状态，确认没有未处理的远端更新或冲突；禁止强推。
- 提交说明要简洁写清本次文章、索引、素材或流程变更范围；提交后向用户说明提交哈希、推送分支和本地是否仍有未提交文件。
