# 后续文章生成方案

> 更新时间：2026-07-15

本文档用于统一后续文章从选题、资料、正文、配图、入库、索引到发布记录的生成方式。当前采用 **方案 A：正文干净化 + 元信息外置**。

## 1. 统一原则

1. **正文只给读者看**：正式文章 Markdown 内不放顶部元信息块，不展示 id、title、description、author、published 等维护字段。
2. **标题只出现一次**：文章标题由文件名、索引和发布平台标题承接，正文内不再重复一级标题。
3. **元信息外置维护**：文章编号、标题、摘要、分类、系列、标签、发布时间、封面、状态等统一写入 `07-资料与流程/文章资产登记表.md`。
4. **先核验再表达**：涉及产品、模型、价格、政策、数据、新闻、官方能力时，先核验资料，再写结论。
5. **图片可追溯**：封面和正文插图统一进入 `08-素材库/图片/文章封面/` 与 `08-素材库/图片/正文插图/`，通过文章编号匹配；来源、许可、访问日期写入素材记录，不放进公开正文。
6. **阅读有缓冲**：每篇正式文章至少安排 1 个正文插图或视觉化版式模块，优先放在一句话结论之后，帮助读者先抓住结构，减少连续纯文本疲劳。
6. **文章即知识库资产**：每篇文章完成后同步分类索引、标签索引、系列索引、发布记录和维护清单。

## 2. 默认交付类型

| 类型 | 适用场景 | 默认产物 | 是否入库 |
| --- | --- | --- | --- |
| 选题卡 | 只要方向、标题、选题池 | 选题、读者收益、差异化、资料边界 | 否 |
| 草稿 | 只要初稿或改写 | 可继续编辑的正文 | 否，除非用户要求 |
| 完整文章 | 要可发布文章 | 正文、标题、摘要、配图建议 | 视用户要求 |
| 知识库入库 | 要持续维护 | 正文、资产登记表、索引、发布记录 | 是 |
| GitHub 发布 | 明确要求提交、推送、同步 | 校验、提交、推送、结果记录 | 是 |

## 3. 标准生成流程

### 3.1 接收需求

先明确以下信息，缺失时可做合理假设，并在交付说明中注明：

- 主题：这篇文章讲什么。
- 目标读者：新手、个人创作者、开发者、团队管理者、企业决策者等。
- 文章类型：工具教程、AI知识、好文方法、安全治理、案例实战、热点追踪。
- 目标动作：读完后希望读者能收藏、实践、转发、关注、购买、部署或复盘什么。
- 资料要求：是否必须查官方文档、论文、新闻、GitHub、产品页。
- 图片要求：是否需要封面、流程图、步骤图、对比图、概念图；若暂不配图，说明使用哪一种视觉化版式模块。
- 发布要求：是否只写草稿，是否入库，是否提交并推送 GitHub。

### 3.2 选题判断

每个选题先过五个问题：

1. 读者是否能立刻知道和自己有什么关系。
2. 是否比泛泛讲概念多一个明确角度。
3. 是否有可核验资料或清楚的经验边界。
4. 是否能给出步骤、清单、模板、案例或判断框架。
5. 是否适合放入现有知识库分类。

### 3.3 Brief 模板

| 项目 | 内容 |
| --- | --- |
| 工作标题 | 文章暂定标题 |
| 一句话定位 | 这篇文章解决什么问题 |
| 目标读者 | 谁最该读 |
| 读者痛点 | 他现在卡在哪里 |
| 核心承诺 | 读完能拿走什么 |
| 文章类型 | 工具教程 / AI知识 / 好文方法 / 安全治理 / 案例实战 / 热点追踪 |
| 资料来源 | 官方文档、项目经验、公开报道、论文等 |
| 事实边界 | 哪些内容必须保守表述 |
| 图片计划 | 封面、流程图、对比图、步骤图等；若暂不配图，写清视觉化版式模块 |
| 入库位置 | 目标文件夹、编号、标签、系列 |

### 3.4 正文结构

#### 工具教程

1. 为什么需要这个工具。
2. 适合谁，不适合谁。
3. 准备工作。
4. 一步一步操作。
5. 常见错误。
6. 最小可复用流程。
7. 下一步建议。

#### AI知识

1. 用一句话解释概念。
2. 它解决了什么旧问题。
3. 它和相邻概念有什么区别。
4. 真实应用场景。
5. 误区和边界。
6. 普通人如何使用。

#### 好文方法

1. 先指出常见误区。
2. 给出新的判断框架。
3. 拆成可执行步骤。
4. 配一个具体例子。
5. 给出复用清单。

#### 安全治理

1. 风险场景。
2. 为什么会发生。
3. 最小安全原则。
4. 权限、数据、审计、验收。
5. 个人和团队清单。

#### 案例实战

1. 背景和目标。
2. 初始问题。
3. 操作过程。
4. 验证方式。
5. 结果和复盘。
6. 可复制模板。

#### 热点追踪

1. 发生了什么。
2. 为什么重要。
3. 对普通用户、开发者、企业分别意味着什么。
4. 哪些信息已确认，哪些仍需观察。
5. 可以提前做什么准备。

### 3.5 Markdown 格式

正式入库文章统一使用以下格式：

```text
> 一句话结论：这篇文章解决什么问题。

## 第一个正文小标题

正文开始。

本文发布前已完成事实、版权与格式检查。
```

禁止：

- 正文顶部放 YAML frontmatter。
- 正文顶部放 HTML 注释元信息。
- 正文里重复 `# 文章标题`。
- 正文末尾附发布前检查清单、参考资料堆叠、图片来源区块、脚注残留、内部复盘信息。
- 在公众号发布版保留 4 列及以上宽表格、`md` / `markdown` / `mermaid` / 无语言代码块。

### 3.6 元信息外置格式

新增文章时，同步在 `07-资料与流程/文章资产登记表.md` 中补充：

| 字段 | 说明 |
| --- | --- |
| 编号 | 与文件名前缀一致 |
| 标题 | 文章正式标题 |
| 分类 | 与目录一致 |
| 系列 | 方便连续阅读 |
| 标签 | 方便检索 |
| 状态 | draft / reviewing / published / archived |
| 发布时间 | 首次发布或入库日期 |
| 更新时间 | 最近一次实质更新日期 |
| 阅读时长 | 估算阅读时间 |
| 文件 | 正文路径 |
| 封面 | 封面路径 |
| 来源核查状态 | 已核对、待复核、需更新等 |

### 3.7 图片方案

- 封面图：`08-素材库/图片/文章封面/编号-文章标题-封面.jpg`。
- 正文插图：`08-素材库/图片/正文插图/编号-文章标题-正文插图01.jpg`。
- 图片来源、许可链接、原图页、访问日期和用途说明写入素材记录或资产登记表；公开正文不写“发布前请复核”。
- 图片应服务文章理解，不做无意义装饰。
- 正文至少配置 1 张本地插图；若主题暂不适合配图，必须用“视觉速读 / 方法卡 / 对比卡 / 检查清单 / 流程卡”等模块替代，并在质量检查中说明。
- 正文插图推荐放在一句话结论之后、第一段正文展开之前，不放在首行，不复用封面图。
- 生成图后检查是否有错字、错误 UI、错误 logo、过度夸张表述。

### 3.8 入库与索引

入库文章必须同步维护：

1. `07-资料与流程/文章资产登记表.md`。
2. `00-知识库导航/知识库总索引.md`。
3. `00-知识库导航/分类索引.md`。
4. `00-知识库导航/标签索引.md`。
5. `00-知识库导航/系列索引.md`。
6. 对应分类目录的 `README.md`。
7. `07-资料与流程/发布记录.md`。
8. `07-资料与流程/内容维护清单.md`。

## 4. 发布前检查

- 正文内没有顶部元信息块。
- 正文第一屏没有重复一级标题。
- 图片链接存在；若没有正文图片，必须存在专门的视觉化版式模块。
- 内部 Markdown 链接存在。
- 资产登记表能查到该文章。
- 分类、标签、系列索引均能回到该文章。
- 涉及最新信息时已核验公开资料。
- 正文没有脚注标记、内部审核记录、图片来源区块、发布前复核提醒和 4 列及以上宽表格。
- GitHub 提交前工作区只包含本次相关改动。

## 5. 完成标准

- 能读：正文干净、结构清楚。
- 能做：读者知道下一步怎么操作。
- 能查：知识库里能通过分类、标签、系列找到。
- 能验：关键事实和图片路径可检查。
- 能更：后续更新能从资产登记表和发布记录继续维护。
## 6. 辅助文件

- `07-资料与流程/文章资产登记表.md`：承接所有文章维护字段。
- `07-资料与流程/后续选题大纲与状态跟踪.md`：管理后续选题池、阶段状态和入库进度。
- `09-工具脚本/封面提示词模板-V1.md`：统一封面生成提示词、配色和后期文字叠加规范。

## 文章质量与微信发布兼容检查

后续新文章默认执行以下质量闸门：

1. 正文不展示元信息表，不重复 H1 标题。
2. 开头统一为一句话结论引用，结尾不写平台 CTA。
3. 加粗默认克制使用，优先通过小标题、引用、列表、卡片和图片突出重点。
4. 发布版正文不保留 `md`、`markdown`、`mermaid`、无语言或嵌套代码块。
5. 发布版正文不保留脚注标记、内部审核记录、图片来源区块或发布前复核提醒。
6. 4 列及以上表格改成卡片、小节或编号清单。
7. Mermaid 先转为 PNG 正文插图，再写入文章。
8. 入库前运行 `09-工具脚本/文章质量扫描.py`，确认视觉化模块、P0/P1 清零后再登记、提交或推送。
