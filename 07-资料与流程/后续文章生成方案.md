# 后续文章生成方案

> 更新时间：2026-07-15
> 适用范围：本仓库后续所有公众号 / AI 自媒体文章的选题、写作、配图、入库、索引和发布维护。
> 目标：每篇文章都能从“一个想法”稳定变成“可阅读、可核验、可入库、可持续更新”的知识库资产。

## 一、统一原则

1. **先定位，再写作**：先确认文章要解决的问题、目标读者、主分类和完成标准，再进入正文。
2. **先核验，再判断**：涉及最新模型、产品、价格、政策、公司动态、论文趋势时，必须先查一手或可信来源，并写清截至日期。
3. **先结构，再润色**：先完成标题、摘要、提纲、核心观点、案例或步骤，再处理语言风格。
4. **文章即知识库资产**：每篇文章必须有编号、分类、标签、系列、隐藏元信息、图片路径、索引记录和发布记录。
5. **读者只看正文**：文章顶部元信息必须隐藏，正文不重复展示一级标题；GitHub 预览中不能出现元信息表格。
6. **图片可追溯**：只保留正式文章实际引用图片，封面和正文插图分目录存放，文件名与文章编号和标题一致。

## 二、默认交付类型

| 类型 | 触发场景 | 默认产物 | 是否入库 |
| --- | --- | --- | --- |
| 选题卡 | 想找方向、热点、系列规划 | 选题列表、评分、适用分类、资料线索 | 否 |
| 文章草稿 | 只要求写初稿、改写、润色 | Markdown 正文草稿 | 否 |
| 完整文章 | 要可发布文章、教程、好文、案例 | 正文、标题、摘要、配图建议、检查清单 | 视用户要求 |
| 知识库入库 | 要分类、索引、持续更新 | 正式文章文件、素材、索引、发布记录 | 是 |
| GitHub 发布 | 明确要求提交、推送、同步 GitHub | 本地校验、提交、推送、提交哈希 | 是 |

> 默认规则：没有明确要求“提交 / 推送 / 同步 GitHub”时，先完成本地文章与知识库维护；明确要求后再提交推送。

## 三、标准生成流程

### 0. 接收需求

先把用户需求归入以下字段：

```text
主题：
目标读者：
文章类型：工具教程 / AI知识 / 好文方法 / 安全治理 / 案例实战 / 热点追踪
目标动作：读完后希望读者会做什么
资料要求：是否需要联网核验；是否必须使用官方来源
图片要求：是否需要封面；是否需要正文插图；是否可用 AI 生成图
发布要求：草稿 / 入库 / 提交 / 推送
```

信息不完整时，优先做合理假设；但涉及真实案例、最新事实、高风险建议、品牌授权、发布动作时必须标记待确认或先核验。

### 1. 选题判断

每个选题按 5 个维度判断：

| 维度 | 判断问题 |
| --- | --- |
| 读者收益 | 读者能少踩什么坑，或马上做成什么事？ |
| 差异化 | 是否避免泛泛讲概念，有没有具体场景 / 方法 / 案例？ |
| 可核验 | 事实、数据、工具能力是否能找到来源？ |
| 可操作 | 是否能给步骤、清单、模板、判断标准？ |
| 可入库 | 是否能归入现有分类、系列和标签？ |

低分选题不直接写长文，先改成“教程”“清单”“案例复盘”或“方法卡”。

### 2. 资料与事实核查

- 最新模型、产品、价格、政策、平台规则、新闻、论文：必须联网核验。
- OpenAI / Codex / ChatGPT / API 相关：优先使用官方文档或官方来源。
- 引用外部观点时：不要大段复制，改用概括，并保留链接或来源名称。
- 事实核查结果写入文章隐藏元信息的 `source_status`，必要时在文末加“资料与核验”。

### 3. 文章 Brief

正式写作前先形成 Brief：

```text
标题方向：
一句话摘要：
核心观点：
读者痛点：
文章结构：
必须出现的步骤 / 清单 / 案例：
不写什么：
需要核验的事实：
需要生成或引用的图片：
```

Brief 不需要总是单独输出给用户，但写作时必须按它组织正文。

### 4. 正文结构

不同文章类型使用不同骨架：

#### 工具教程

```text
开头：读者要完成什么任务
适用场景：什么时候该用 / 不该用
准备条件：账号、权限、文件、环境
步骤 1-N：每步有结果，不只写概念
常见错误：至少 3 个
检查清单：读者如何确认做成了
下一步：如何复用到自己的工作流
```

#### AI 知识

```text
开头：用一句话解释概念解决什么误解
是什么：用生活化比喻，但不牺牲准确性
为什么重要：连接真实工作场景
怎么判断：给边界和反例
怎么应用：给简单做法或清单
延伸阅读 / 核验来源
```

#### 好文方法

```text
开头：指出低效做法
新方法：给一个可复用框架
步骤：拆成可执行动作
例子：用具体任务演示
模板：给读者可复制内容
结尾：给下一次行动，而不是喊口号
```

#### 安全治理

```text
开头：指出风险不是恐吓，而是边界管理
风险场景：哪些数据、权限、流程容易出问题
判断标准：什么能交给 AI，什么不能
操作清单：权限、脱敏、复核、留痕
团队规则：如何写进 AGENTS.md / SOP / 审批流
```

#### 案例实战

```text
背景：任务目标、限制和成功标准
过程：怎么拆解、怎么让 AI 执行
关键决策：为什么这么做，不做什么
结果：完成了什么，如何验证
复盘：可复用经验和风险
```

#### 热点追踪

```text
事件：发生了什么，时间和来源
影响：对普通用户 / 创作者 / 企业分别意味着什么
误区：哪些说法可能被夸大
行动建议：现在该观察、尝试还是暂缓
截至日期与来源
```

### 5. Markdown 格式

正式文章使用隐藏元信息块：

```markdown
<!--
---
id: "35"
title: "中文文章标题"
description: "一句话说明文章解决什么问题"
author: "Codex Article Lab"
published: "2026-07-15"
updated: "2026-07-15"
category: "工具教程"
series: "Codex工作流"
tags: ["Codex", "AI工作流"]
status: "published"
reading_time: "8 分钟"
source_status: "已核验官方资料；建议后续更新时重新核验"
cover: "../08-素材库/图片/文章封面/35-中文文章标题-封面.jpg"
---
-->
```

正文要求：

- 不再写 `# 中文文章标题`。
- 第一屏优先出现核心观点、封面图或 `##` 小标题。
- 小标题使用 `##` / `###`。
- 普通段落尽量短，适合手机阅读。
- 表格只在信息确实需要对比时使用。
- 代码块、命令、链接必须能复制或能打开。

### 6. 图片方案

| 图片类型 | 存放位置 | 命名 |
| --- | --- | --- |
| 文章封面 | `08-素材库/图片/文章封面/` | `NN-中文文章标题-封面.jpg` |
| 正文插图 | `08-素材库/图片/正文插图/` | `NN-中文文章标题-正文插图01.png` |

图片使用规则：

1. 优先真实截图、官方素材、用户提供素材。
2. 没有合适素材时，再使用 AI 生成图。
3. AI 生成图要服务文章理解，不做无意义装饰。
4. 图片进入仓库前必须确认文章中确实引用；未引用图片不长期保留。
5. 文章路径使用相对路径，例如：`../08-素材库/图片/文章封面/NN-标题-封面.jpg`。

### 7. 入库与索引

正式入库时必须同步：

- 根目录 `README.md`
- `00-知识库导航/知识库首页.md`
- `00-知识库导航/知识库总索引.md`
- `00-知识库导航/分类索引.md`
- `00-知识库导航/系列索引.md`
- `00-知识库导航/标签索引.md`
- 对应分类目录的 `README.md`
- `07-资料与流程/发布记录.md`
- `07-资料与流程/内容维护清单.md` 或本次检查记录

索引更新规则：

- 按文章编号升序排列。
- 保留已发布文章编号，不重编号。
- 主分类只选一个；跨主题用标签和系列解决。
- 移动文章时同步修复所有内部链接。

### 8. 发布前检查

发布前必须通过：

- [ ] 文章编号未重复。
- [ ] 文件位于正确中文分类目录。
- [ ] 隐藏元信息块完整，GitHub 预览不显示元信息表格。
- [ ] 正文没有重复一级标题。
- [ ] 封面和正文插图路径存在。
- [ ] 内部链接可打开。
- [ ] 涉及最新事实的内容已核验并记录来源。
- [ ] README、总索引、分类索引、系列索引、标签索引已同步。
- [ ] 发布记录已新增或更新。
- [ ] 提交前查看改动范围，确认没有误删素材或历史文章。

## 四、完成标准

一篇文章只有同时满足以下条件，才算完成：

1. **能读**：开头明确，结构清楚，移动端阅读不累。
2. **能做**：读者能按步骤、清单或模板执行。
3. **能查**：文章在总索引、分类索引、系列索引、标签索引中能找到。
4. **能验**：事实、图片、链接和代码块可核查。
5. **能更**：后续知道文章编号、分类、素材位置和更新记录。

## 五、推荐使用方式

后续可以直接这样下任务：

```text
按《后续文章生成方案》生成一篇文章：
主题：
类型：工具教程 / AI知识 / 好文方法 / 安全治理 / 案例实战 / 热点追踪
目标读者：
读完要会做什么：
是否需要联网核验：是 / 否
是否需要配图：封面 / 正文插图 / 不需要
是否入库并推送 GitHub：是 / 否
```

如果只是想要草稿，可以说“只生成草稿，不入库不提交”。如果要完整维护仓库，可以说“文章入库 + 分类 + 索引 + 发布记录 + 推送”。
