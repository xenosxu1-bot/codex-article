# 选题同步与规划管控机制

> 更新时间：2026-07-16  
> 作用：把“选题规划、查重、写作、质检、入库、同步”的动作固定下来，保证 main 作为个人私密工作区，prod 作为可共享知识库时不混乱。

## 一、仓库与分支定位

| 分支/仓库 | 定位 | 可以包含 | 不建议包含 |
| --- | --- | --- | --- |
| main | 个人生产与规划工作区 | 选题库、草稿、内部复盘、工具脚本、生成记录、待确认素材 | 如果 GitHub 仓库是公开的，则不要放真正私密内容。 |
| prod | 共享知识库展示区 | 已确认可公开的文章、导航索引、素材、发布记录、读者可用流程 | 未公开选题、内部审核细节、未确认版权素材、半成品草稿。 |

> 重要提醒：如果 main 和 prod 是同一个 GitHub 仓库的不同分支，分支本身不是隐私边界。真正需要私密时，应使用“私有源仓库 + 公开/共享发布仓库”的双仓库模式。

## 二、从选题到发布的固定流程

```text
灵感收集
  ↓
选题库登记
  ↓
已发布文章去重
  ↓
评分与排期
  ↓
Brief 与资料边界
  ↓
正文生成/修订
  ↓
文章质量检查
  ↓
资产登记与索引更新
  ↓
main 提交
  ↓
用户确认
  ↓
同步 prod
  ↓
发布后复盘
```

## 三、每次新增选题的管控清单

- [ ] 已在 `选题库.md` 登记选题ID、分类、系列、目标读者、核心问题、交付物。
- [ ] 已在 `已发布文章去重登记.md` 查重。
- [ ] 若与旧文相近，已写明新增角度。
- [ ] 已按 5 个维度评分并标注 P0/P1/P2。
- [ ] 已确认是否需要外部资料核查。
- [ ] 已确认是否适合进入近期排期。

## 四、写作前 Brief 必填项

1. 目标读者：这篇到底写给谁。
2. 核心问题：读者为什么需要这篇。
3. 不写什么：明确边界，避免泛化。
4. 核心观点：全文只围绕一个主张展开。
5. 交付物：模板、清单、步骤、流程图或案例复盘。
6. 资料要求：是否需要联网核查、官方文档、截图或来源链接。
7. 风险点：事实、版权、隐私、安全、品牌表达、过度承诺。
8. 关联旧文：可以引用哪些已发布内容，避免重复哪些内容。

## 五、main 到 prod 的同步门槛

只有同时满足以下条件，才建议同步到 prod：

- [ ] 正文不含顶部元信息表、不重复一级标题。
- [ ] Markdown 段落、标题、列表、代码块、引用块能正常转微信 HTML。
- [ ] 图片已入素材库并在图片生成记录或版权记录中可追踪。
- [ ] 表格已评估移动端阅读体验；长表格已转为卡片或小节。
- [ ] 脚注、来源、外部链接已核查；无孤立脚注标记。
- [ ] 内部审核记录已删除或改成公开读者可接受的说明。
- [ ] `文章资产登记表.md`、索引、发布记录已同步。
- [ ] 用户已明确确认可同步 prod。

## 六、节奏建议

| 周期 | 动作 | 输出 |
| --- | --- | --- |
| 每次写作前 | 查重、评分、Brief | 一条可写作选题 |
| 每周 | 复盘选题库 P0/P1，移动状态 | 本周写作排期 |
| 每月 | 清理 HOLD/DROP，检查系列缺口 | 下月主题规划 |
| 每次发布后 | 更新资产登记、去重登记、发布记录 | 可追踪的知识库条目 |
| 每次同步 prod 前 | 执行公开发布检查 | 可共享知识库版本 |

## 七、冲突处理规则

- **标题冲突**：优先保留已发布标题，新选题改为更具体的场景标题。
- **主题冲突**：如果同一核心问题已经被旧文解决，新选题必须改成“案例版、模板版、团队版、更新版”之一。
- **素材冲突**：同一封面或插图不要重复使用到不同文章，除非是系列统一视觉模板。
- **分支冲突**：后续修改默认先落 main；prod 只接受确认后的同步，不在 prod 上直接规划新内容。

## 八、可沉淀到 Skill / AGENTS.md 的规则

- `article-wechat Skill`：记录“选题入库 → 去重 → Brief → 质检 → 资产登记 → 索引更新 → 发布记录”的完整流程。
- `AGENTS.md`：记录本项目的默认约定：新增文章前必须查重，正式文章不保留顶部元信息，main 先行、prod 后同步。
- 如果未来要自动提醒每周复盘选题库，可建立 Codex Automation。
