# article-wechat

用于交付可发布的中文公众号文章和内容生产包，而不是只给一段泛文案。默认遵循：事实可核验、图片可追溯、格式适合移动端、发布动作可回滚、入库结构可持续维护。

## 启动顺序

1. 先读取 `references/recurring-rules.md`。
2. 判断任务模式，并在交付中说明关键假设。
3. 按需读取参考文件：
   - 输出契约：`references/output-contract.md`
   - 完整流程：`references/ai-media-production-system.md`
   - 文章生成方案：`references/article-generation-playbook.md`（用户要求统一后续文章生成方案、按方案写文章、生成新文章或沉淀文章流程时必读）
   - 全自动生产工作流：`references/automation-workflow.md`（用户要求完整生产包、全自动文章生产、批量图文包或整改生产流程时必读）
   - 选题与系列规划：`references/topic-selection-and-series-planning.md`（用户要求热门选题、选题库、内容规划、栏目或连续发文时必读）
   - 写作标准：`references/editorial-standards.md`
   - 去 AI 化检查：`references/de-ai-writing-checklist.md`（AI 方法文、AI 工具文、Agent 实战和审稿改写时必读）
   - 结构承接检查：`references/structure-cohesion-checklist.md`（完整生产包、深度稿和审核修订时必读）
   - 事实核查：`references/fact-check-policy.md`
   - 视觉与版权：`references/visual-policy.md`
   - 正文插图分镜：`references/image-storyboard-workflow.md`（需要正文插图、流程图、卡片图或图文包时必读）
   - 质检清单：`references/qa-checklist.md`
   - GitHub 发布：`references/publishing-github.md`
   - 文章知识库维护：`references/article-knowledge-base-workflow.md`（用户要求文章入库、分类、索引、发布记录、知识库维护、持续更新或整理仓库时必读）
   - 正文模板：`references/wechat-article-template.md`
4. 交付前完成内容真实性与基本格式核验；正文不附发布前检查或参考资料区块。

## 规则优先级

如文档之间出现冲突，按以下顺序执行：用户本次明确指令 > `SKILL.md` > `references/recurring-rules.md` > 对应专项文档（如 `article-knowledge-base-workflow.md`、`publishing-github.md`、`output-contract.md`）> 旧版参考文档。发布触发条件以本文件和 `references/publishing-github.md` 为准。

## 任务模式

| 模式 | 触发 | 默认输出 | 是否发布 |
| --- | --- | --- | --- |
| 仅选题 | 用户要选题、方向、栏目 | 5-10 个选题卡和评分 | 否 |
| 仅草稿 | 用户要写一篇草稿、初稿、改写 | Markdown 正文，可附标题候选 | 否 |
| 完整生产包 | 用户要完整文章、可发布包、图文包、全自动生产 | 正文、图片、结构地图、断言台账、标题评分、去 AI 化报告与真实性核验结果 | 否 |
| 审核修订 | 用户提供已有文章要求检查、优化 | 问题清单、修订建议、必要时给修订稿 | 否 |
| 知识库维护 | 用户要求文章入库、分类、索引、发布记录、整理仓库、持续更新 | 中文分类目录、总索引、分类/系列/标签索引、发布记录、维护清单 | 按用户要求；若涉及推送则是 |
| 发布到 GitHub | 用户明确说发布、上传、推送、同步到 GitHub | 编号、入库、索引更新、提交、推送、发布记录 | 是 |

**发布采用模式 C：只有用户明确要求“发布/上传/推送/同步到 GitHub”时，才执行编号、入库、更新索引、提交和推送。** 单纯“写文章、做草稿、做完整生产包”不触发发布。

## 完整生产包默认流水线

用户要求完整文章、可发布包、图文包、全自动生产或批量生产时，默认执行 `references/automation-workflow.md`：选题去重与评分 → Brief → 来源台账 → 标题评分 → 结构地图 → 初稿 → 关键断言核查 → 去 AI 化改写 → 封面与正文插图 → 格式检查 → 独立 QA → 生产包归档。P0/P1 未清零不得标记为可发布。

完整生产包必须至少包含：`README.md`、`brief.md`、`titles.md`、`structure-map.md`、`source-ledger.md`、`claim-ledger.md`、`fact-check.md`、`cover-brief.md`、`inline-image-storyboard.md`、`image-ledger.md`、`de-ai-report.md`、`format-report.md`、`qa-report.md`、`publish-record.md` 和本地图片目录。只有用户明确要求发布、上传、推送或同步到 GitHub 时，才继续执行发布流程。

## 信息缺失处理

可以合理假设：目标渠道为微信公众号、语气为有判断有边界的中文创作者口吻、默认交付为本地 Markdown 草稿或生产包。

必须询问或标记待确认：是否发布到 GitHub、是否使用真实图片或 AI 视觉、是否涉及医疗/法律/金融等高风险建议、是否使用未公开资料、是否使用第一人称真实经历或测试结果。

## 不可违反的底线

- 不虚构事实、数据、客户案例、采访、亲身测试或第一人称经历。
- 涉及最新工具、模型、价格、法规、人物、新闻或平台规则时，必须核验一手来源，并写清截至日期。
- 事实、推断、观点和假设必须分开表达。
- AI 视觉必须标注，不伪装成实拍、官方图、产品截图或真实案例。
- 图片必须本地可引用，并保留替代文本、图注、来源和授权说明。
- 标题不得捏造数字、制造恐慌、过度承诺，必须与正文一致。
- 交付前必须完成内容真实性核验；正文不附检查清单、参考资料或来源列表。
- 不把账号登录信息、密钥、Cookie、未公开资料或读者隐私写入 Skill、提示词或仓库。

## 正文格式硬规则

- 正文不使用中文全角引号；需要突出时使用 Markdown 加粗，例如 `**重点内容**`。
- 正文不使用 Markdown 分割线。
- 标题默认不使用问号，除非用户明确要求保留。
- 正文不设置发布前检查、参考资料、参考文献、资料来源或延伸阅读等区块。事实核验在内部完成，正文只保留经核验的结论与必要边界。

## 封面与配图生成默认流程

当用户要求生成、优化或批量制作文章封面、正文插图或配图时，先读取 `references/visual-policy.md` 和 `references/templates/cover-prompt-template.md`；如果当前项目存在 `09-工具脚本/封面提示词模板-V1.3.md`，以项目内 V1.3 作为当前执行版本，V1/V1.2 仅作为历史参考。生成前必须先执行风格规范检查：比例、标题副标题安全区、主题视觉、关键词卡片、文字量、品牌/水印/乱码风险、移动端缩略图可读性。需要正文插图时，先生成正文插图分镜表，明确每张图服务哪一节、解决什么理解问题、放置位置、图前承接句和图后推进句。

默认封面系统采用 V1.3：2.35:1 横版，标准画布 2350 × 1000 px，微信公众号优先输出 1792 × 764 px；左侧保留 55%—60% 干净标题区，只放主标题和副标题，不添加第三行价值句、分割线、横线或穿过文字的光轨；右侧只保留 2—3 个与文章主题直接相关的原创视觉元素；底部放 4 个能力关键词卡片，图标、关键词和短说明在卡片内居中。整体为深色 AI 科技产品发布会风格，按文章主题动态调整配色。

正式封面默认使用“AI 视觉底图 + 本地确定性中文叠字”的两阶段方式，避免 AI 生成中文错字。图片模型只负责背景、右侧主体、图标化模块、光效和空间层次；中文主标题、副标题、短标签和最终成品由本地脚本完成。封面文字必须少而强：主标题优先压缩为 8—16 个中文字符，最多 2—3 行；副标题 12—22 个中文字符；图标标签 3—5 个，每个 2—4 字。禁止长句、密集小字、说明型信息卡、真实 Logo、真实产品 UI、人物肖像、水印、二维码、品牌复制和 AI 乱码。

### 当前项目作图脚本识别

在文章知识库项目中，优先查找并使用 `09-工具脚本/` 下的脚本：

- `gpt_image_cover.py`：默认单篇封面生成入口。支持 `--title`、`--subtitle`、`--tags`、`--theme`、`--palette`、`--output-size`、`--stem`、`--skip-api`；适合单篇封面、重做封面、只预览本地排版或检查风格参数。
- `批量补全文章封面.py`：默认批量补缺封面入口。支持 `--only 编号...` 和 `--force`；适合根据文章资产登记表批量补图。`--force` 会重做已有封面，必须在用户明确确认后使用。
- `批量生成配图.py`：历史正文图/配图生成脚本，依赖固定素材库文件，且可能没有安全的 `--help` 入口；不要作为默认批量入口直接运行。执行前必须先核对依赖素材是否存在、输出目录是否正确，并保留原图备份。

默认输出与记录位置：

- 封面底图：`08-素材库/图片/封面底图/`
- 文章封面：`08-素材库/图片/文章封面/`
- 正文插图：`08-素材库/图片/正文插图/`
- 图片生成记录：`07-资料与流程/图片生成记录.md`
- 文章封面引用：同步到 `07-资料与流程/文章资产登记表.md`、对应文章正文和 README/索引（如项目规则要求）。

### 封面提示词模板工作流

1. 读取文章标题、摘要、分类、系列、标签和读者场景，先提炼封面主标题、副标题、3—5 个图标标签、右侧视觉主体和主题配色。
2. 按 V1.3 模板生成“无文字或仅占位文字”的底图提示词，明确要求 no readable words、no logo、no watermark、no real UI、no portrait，并保留左侧干净标题安全区。
3. 选择配色：`dev` 用于 AI 编程/Agent/自动化，`security` 用于安全/权限/治理，`content` 用于内容创作，`research` 用于知识管理/研究，`business` 用于商业/团队/ROI。
4. 单篇封面优先使用：`python 09-工具脚本/gpt_image_cover.py --title "封面主标题" --subtitle "封面副标题" --tags "标签1,标签2,标签3,标签4" --theme "右侧视觉主体" --palette dev`。只想检查本地排版或不调用图片模型时加 `--skip-api`。
5. 批量补缺封面优先使用：`python 09-工具脚本/批量补全文章封面.py --only 26 28`；只有用户明确要求全部重做时，才使用 `--force`。
6. 不读取、不展示、不提交 `.env` 中的密钥内容；只提示需要配置 `OPENAI_API_KEY`、`OPENAI_BASE_URL`、`OPENAI_IMAGE_MODEL=gpt-image-2`、`OPENAI_IMAGE_SIZE`、`OPENAI_IMAGE_QUALITY`。
7. 生成后逐张质检：尺寸比例、左侧安全区、手机缩略图可读性、标题/副标题无错字、右侧元素不遮挡标题、底部卡片居中、无真实 Logo/水印/二维码/人物、无 AI 乱码。
8. 质检通过后更新图片生成记录、文章资产登记表、文章正文图片引用和知识库索引；只有用户明确要求推送/同步时，才提交并推送 GitHub。

### 正文配图工作流

深度教程或案例文章默认至少配置 2 张正文图，但不把封面重复放入正文。正文图优先使用流程图、步骤图、对比卡片、检查清单或可视化摘要；4 列及以上表格优先转为移动端友好的卡片/小节。所有正文图保存到 `08-素材库/图片/正文插图/`，并在正文中用相对路径引用，配上可读 alt 文本和简短图注。涉及官方截图、第三方素材或 AI 生成图时，版权、来源、访问日期和生成说明写入内部记录，不在公开正文堆叠审核区块。

### 无人物产品工作台封面风格

当用户要求封面贴近深色 AI 科技发布会、Codex/OpenAI 工具海报、AI 技术公众号头图，或反馈人物、AI 头像、数字人不够高级时，默认切换为**无人物产品工作台风格**：不出现真人、背影人物、数字人头像、拟人机器人或人脸轮廓；右侧用原创软件工作台、仪表盘、代码/资料侧栏、插件卡、Skill 卡、Workflow 流程、发光立方体或终端核心来表达主题。

该风格的默认构图为：左侧保留 42%—45% 的清晰标题区，主标题 2—3 行，副标题 1 行；如果标题较长或用户明确要求极简留白，则左侧安全区扩展到 55%—60%。右侧 55%—58% 为主题视觉，包含 1 个主产品界面、2—3 个侧边代码/资料/任务面板、1 个低调发光核心物体。底部只放 4 个能力关键词卡片，图标与关键词在框内居中；短说明可选，必须小而克制。生成前必须先检查：无人物、无真实 Logo、无可读小字、无横线穿过文字、右侧元素不遮挡标题、底部卡片居中。

知识库维护任务默认包括：确认目标仓库与分支、拉取远端最新状态、识别下一篇文章编号、选择主分类、补充 frontmatter 元数据、移动或新增文章和图片、更新总索引/分类索引/系列索引/标签索引、更新发布记录和维护清单、检查本地链接和图片路径。只有用户明确要求发布、推送或同步到 GitHub 时，才执行提交和推送。

## 后续文章生成默认流程

1. 先按 `references/article-generation-playbook.md` 建立 Brief：主题、目标读者、文章类型、目标动作、资料要求、图片要求、发布要求。
2. 再写正文：不同类型文章使用对应骨架；正文不重复展示一级标题。
3. 正式文章正文不保留顶部元信息块；编号、标题、摘要、分类、标签、封面、状态等维护字段统一写入项目内 `07-资料与流程/文章资产登记表.md`。

5. 如需提交或推送，继续执行 `references/publishing-github.md`，先校验再提交，不强推。

## 知识库导航分层

执行文章入库、分类、索引、发布记录或知识库维护任务时，按以下口径判断：

- 分类 = 主书架：每篇文章只选一个主分类，并与正文所在中文目录一致。
- 系列 = 阅读路径：可以跨分类，用于连续教程、专题学习、项目复盘和新手路线。
- 标签 = 检索词：每篇建议 3-5 个，优先使用稳定、可复用、可跨分类的关键词。
- 总索引 = 全量清单：用于核对编号、路径、分类、系列、标签和状态。
- 文章资产登记表 = 唯一元数据来源：所有导航索引以登记表同步，不把元信息放回正文顶部。

读者入口优先首页、系列和分类；标签作为二级检索入口；发布记录、维护清单和资产登记表属于后台维护入口。

## 开源 AI 工具教程生产规则

- 写 GitHub 开源 AI 工具、Agent、模型框架或开发者工具教程时，默认先核验官方 GitHub README、官方文档、Quickstart、安装说明、版本文件和安全/权限相关说明；在内部素材或发布记录中写明访问日期、仓库链接、文档链接和截图来源，不把参考资料区块放进正文。
- 正文默认按新手最小闭环组织：项目定位 → 安装前准备 → 安装命令 → 模型或账号配置 → 跑通第一次可验证任务 → 会话恢复或结果验证 → 再扩展插件、Skills、MCP、Gateway、自动化或团队能力。
- 工具教程必须区分已核验事实、基于官方文档的推断和作者建议；不得虚构亲测数据、效率倍数、客户案例、生产环境经历或未验证功能。
- 新手快速上手类文章至少提供 1 张封面、2 张正文图；优先使用官方真实界面截图或官方仓库随附图片。若官方只有英文界面，正文用中文解释关键区域；若使用自制流程图或 AI 视觉，图注必须标明类型，禁止伪造成官方截图。
- 正文插图必须直接插入相关段落之间，不保留裸图片地址、素材清单、下载链接或 HTML 图片标签。图片文件统一本地化到 `08-素材库/图片/正文插图/`，文件名与文章编号和主题一致。
- 教程中出现命令时，优先使用短行内命令或编号步骤；微信公众号发布版避免大段围栏代码块。安装命令、登录命令、恢复命令和排错命令必须来自官方文档或仓库说明。
- 每篇开源工具教程至少给读者一个可复用资产，例如 30 分钟练习路线、最小任务卡、风险清单、验收标准或排错顺序，让读者读完能直接执行。
- 结尾不写参考资料、发布前检查和平台 CTA；用下一步行动、适用边界或新手避坑自然收束。

## 深度文章颗粒度与图文要求

- 对 AI 工具使用、AI 学习教程、工作流、Agent 实战、案例复盘类长文，默认按深度稿交付：开头一句话结论后，正文至少覆盖背景问题、任务卡或输入模板、流程拆解、执行清单、验收标准、风险边界、复用模板或下一步行动。
- 每篇深度稿至少配置 2 张正文图：1 张流程图或架构图解释方法路径，1 张贴图或卡片总结检查项、能力关键词或验收口径；正文图不替代封面图，不放在正文首行。
- 文中图片必须本地化到 `08-素材库/图片/正文插图/`，文件名与文章编号和主题一致；图注标明自制示意图、AI 视觉或真实截图类型。
- 正文图片必须直接插入到相关文字段落之间，只保留可渲染的 Markdown 图片语法和简短图注；禁止把图片地址、HTML `<img src=...>`、不存在的本地路径或素材清单当正文内容保留。微信公众号发布版需要在编辑器中上传/插入图片，不能让读者看到本地图片路径。
- 若使用第一人称测评、实战或复盘标题，正文必须说明是可复现实验、方法模板或真实已核验经历，不虚构测试、客户案例、数据或亲身经历。
- 写作时避免只有概念解释；每个核心观点尽量配一个可执行动作、提示词、清单、判断标准或验收口径。
## AI 文章去内容生成口吻规则

- AI 方法文、AI 工具文和 Agent 实战文章先写具体场景，再解释概念；避免用宏大背景、趋势判断或术语堆叠开头。
- 每个抽象 AI 概念至少配一个生活或工作类比，再给正式解释，确保读者能先理解“像什么、解决什么”。
- 每篇 AI 深度稿至少提供一个可复用资产，例如任务卡、检查表、流程图、决策表或提示词模板，让读者读完能直接执行。
- 禁止无证据的效率倍数、亲测经历、客户案例和绝对化承诺；如标题或正文涉及测试/经验，必须有可复现实验条件或明确边界。

## 文章质量与微信发布兼容规则

- 正文只保留可发布内容，不展示元信息表、frontmatter 或隐藏元数据块；元信息外置到文章资产登记表、索引和发布记录。
