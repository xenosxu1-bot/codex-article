# 写作与编辑标准

## 文章类型分流

| 类型 | 重点 | 主要风险 |
| --- | --- | --- |
| 热点解读 | 时效、来源、判断边界 | 事实过期、跟风、过度推断 |
| 工具教程 | 步骤、截图/示例、可复现 | 功能变化、夸大效果 |
| 方法清单 | 场景、步骤、取舍 | 空泛、不可执行 |
| 案例拆解 | 过程、证据、限制 | 虚构案例、幸存者偏差 |
| 观点评论 | 立场、事实依据、反方观点 | 情绪化、证据不足 |
| 转化内容 | 服务边界、适用对象、CTA | 广告伪装客观评测 |
| 复盘文章 | 背景、链路、根因、改进 | 归因泛化、只追责不改机制 |

## 正文要求

- 使用一个 H1；保持 H2 -> H3 层级，不跨级。
- 三段内给出具体场景、读者问题、核心结论或张力。
- 每节只推进一个观点；用具体动作、例子、清单或表格降低理解成本。
- 普通段落以 1-3 句为主；移动端难读的长表格改成分点清单。
- 结尾给出下一步动作，但不强迫、不制造焦虑。
- 正文不使用中文全角引号；需要突出时使用 Markdown 加粗。
- 正文不使用 Markdown 分割线。
- 正文不附发布前检查、参考资料、参考文献、资料来源或延伸阅读等区块；内容真实性在写作流程内完成核验。

## 文风

默认使用有判断、有边界的中文创作者口吻：具体、克制、可执行。删除没有证据支撑的“颠覆、赋能、重塑、全网最强、彻底改变”等空词。

## 标题

- 生成至少 8 个标题，覆盖利益型、问题型、反常识型、案例型。
- 按准确性、具体性、读者收益、辨识度和不过度承诺筛选。
- 标题承诺的价值必须在正文前 30% 得到回应。
- 不捏造数字，不制造恐慌，不使用与正文不符的悬念。
- 默认不使用问号；只有用户明确要求时才保留问号。

## 文章质量与微信发布兼容规则

- 正文只保留可发布内容，不展示元信息表、frontmatter 或隐藏元数据块；元信息外置到文章资产登记表、索引和发布记录。
- 正文不重复文章标题，不使用 H1；开头默认使用 `> 一句话结论：……`。
- 封面图外置到素材库和文章资产登记表，不作为正文第一张图重复展示。
- 加粗只用于结论、风险、行动建议和关键限制条件；每篇建议不超过 6 处，超过 10 处视为发布前 P1 问题。
- 禁止 `****`、相邻加粗、整句加粗、单字加粗。
- 微信公众号发布版避免 `md`、`markdown`、`mermaid`、无语言和嵌套代码块；Mermaid 先转 PNG，再作为正文插图引用。
- 结尾使用自然收束或下一步建议，不把“点在看”“关注公众号”等平台 CTA 写入知识库正文。
- 发布或提交前运行项目内文章质量扫描，确认 P0/P1 清零。
