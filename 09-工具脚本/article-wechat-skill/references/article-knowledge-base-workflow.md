# 文章知识库维护流程

用于把公众号 / AI 自媒体文章从“单篇文件”维护成可长期查阅、更新和发布的中文知识库。

## 触发条件

当用户出现以下表达时启用本流程：

- 文章入库
- 分类整理
- 更新索引
- 发布记录
- 知识库维护
- 仓库文章编排
- 方便后续持续更新
- 把文章整理成知识库
- 扫描仓库所有文件并分类

如果用户只要求写草稿、改写、审核文章，不默认执行知识库维护；除非用户明确要求入库、整理、同步或发布。

## 默认目标

1. 文章可持续更新：新增文章有固定编号、分类、标签和状态。
2. 知识库可查：读者可以从 README、知识库首页、总索引、分类索引、系列索引、标签索引进入。
3. 素材可追溯：图片、封面、发布包、脚本分区存放，文章引用路径稳定。
4. 发布可回滚：提交前知道改了哪些文件；推送前确认远端状态；不强推、不覆盖。
5. 规则可复用：把目录约定、检查清单和“完成标准”写进项目 `AGENTS.md` 或维护文档。

## 推荐中文目录结构

若项目尚无明确约定，优先采用以下结构；若项目已有 `AGENTS.md` / `README.md` 约定，以项目约定为准。

```text
00-知识库导航/
01-工具教程/
02-AI知识/
03-好文方法/
04-安全治理/
05-案例实战/
06-热点追踪/
07-资料与流程/
08-素材库/
  图片/
09-工具脚本/
README.md
AGENTS.md
```

说明：`README.md` 和 `AGENTS.md` 可以保留英文名，因为 GitHub 和 Codex 会优先识别它们；其他面向读者的文件夹和文件名尽量使用中文。

## 文章命名规则

- 正式文章使用：`NN-中文标题.md`
- `NN` 使用两位数编号，例如 `01`、`08`、`34`。
- 新文章使用下一个未占用编号。
- 已发布文章原则上不重编号，避免旧链接和引用失效。
- 文章移动分类时保留原编号。

## 分类原则

每篇文章只能有一个主分类，避免重复存放；跨主题关系通过标签、系列索引和总索引解决。

推荐主分类：

| 分类 | 适用内容 |
| --- | --- |
| 工具教程 | Codex、ChatGPT、插件、MCP、自动化、图像生成、文档处理等上手教程 |
| AI知识 | 模型、Agent、RAG、提示词、上下文、AI 原理和概念解释 |
| 好文方法 | 写作方法、工作流方法论、效率方法、选题与内容运营 |
| 安全治理 | 隐私、安全、合规、风险、版权、事实核查、企业治理 |
| 案例实战 | 项目复盘、真实任务拆解、端到端实操案例 |
| 热点追踪 | AI 新闻、热门工具、行业事件、趋势观察；必须标明截至日期和来源 |
| 资料与流程 | 维护规则、发布记录、选题池、检查清单、工作流文档 |
| 素材库 | 文章封面、正文插图和必要附件；不长期保留整包发布归档 |
| 工具脚本 | 辅助生成、检查、整理和发布的脚本 |

## 推荐隐藏元信息块

正式文章建议包含以下元数据，方便检索和后续自动化处理；必须使用 HTML 注释隐藏，避免 GitHub/Markdown 预览展示成表格：

```markdown
<!--
---
id: 35
title: 中文标题
description: 一句话摘要
author: Codex
published: 2026-07-15
updated: 2026-07-15
category: 工具教程
series: Codex工作流
tags:
  - Codex
  - AI工作流
status: published
reading_time: 8min
source_status: verified
cover: ../08-素材库/图片/文章封面/35-中文文章标题-封面.png
---
-->
```

字段说明：

- `id`：文章编号，和文件名前缀一致。
- `category`：主分类，必须和所在目录一致。
- `series`：可为空，但建议用于连续内容。
- `tags`：用于标签索引，不作为主分类。
- `status`：可用 `draft`、`review`、`published`、`archived`。
- `source_status`：可用 `unchecked`、`needs-review`、`verified`。
- `cover`：使用相对路径，指向素材库中的本地图片。

## 与文章生成方案的关系

新文章先按 `article-generation-playbook.md` 完成选题、Brief、正文、配图和检查；确认入库后，再执行本文件的编号、分类、索引和发布记录流程。

## 入库流程

1. **确认仓库状态**
   - 确认目标仓库、分支和远端地址。
   - 先获取远端最新状态。
   - 确认本地是否有未提交改动；不要覆盖用户已有改动。

2. **确认编号**
   - 读取 `00-知识库导航/知识库总索引.md` 和现有文章文件。
   - 找到下一个未占用编号。
   - 如果文章已经发布过，保留原编号，不重编号。

3. **确认分类**
   - 按文章主要用途选择一个主分类。
   - 不确定时优先选择读者最可能查找的分类。
   - 跨分类属性写入 `tags` 或 `series`。

4. **整理文章文件**
   - 文件名使用 `NN-中文标题.md`。
   - 放入对应中文分类目录。
   - 补充或规范隐藏元信息块。
   - 检查标题、摘要、标签、图片路径、来源说明；正文不要重复展示一级标题。

5. **整理素材**
   - 图片放入 `08-素材库/图片/`，文件名使用“`NN-中文文章标题-用途`”：封面为 `文章封面/NN-中文文章标题-封面.扩展名`，正文图为 `正文插图/NN-中文文章标题-正文插图01.扩展名`。
   - 不长期保留整包发布归档；只保留正式文章实际引用的图片素材。
   - 脚本放入 `09-工具脚本/`。
   - 文章中图片路径使用相对路径，例如 `../08-素材库/图片/xx.png`。

6. **更新索引**
   - 更新根目录 `README.md`。
   - 更新 `00-知识库导航/知识库首页.md`。
   - 更新 `00-知识库导航/知识库总索引.md`。
   - 更新 `00-知识库导航/分类索引.md`。
   - 更新 `00-知识库导航/系列索引.md`。
   - 更新 `00-知识库导航/标签索引.md`。
   - 如分类目录有 README，同步更新分类内文章清单。

7. **更新记录**
   - 更新 `07-资料与流程/发布记录.md`：记录日期、编号、标题、分类、改动摘要、提交哈希或待提交状态。
   - 更新 `07-资料与流程/内容维护清单.md`：记录待复查、待补图、待事实核查、待重写等事项。
   - 重大结构调整时更新 `00-知识库导航/旧链接迁移说明.md`。

8. **检查质量**
   - 文章编号连续且无重复。
   - 本地 Markdown 链接有效。
   - 本地图片链接有效。
   - 代码块闭合。
   - 无明显乱码。
   - 根目录没有散落文章。
   - 图片、引用和 AI 视觉说明符合版权与事实核查规则。

9. **提交与推送**
   - 只有用户明确要求发布、推送或同步到 GitHub 时执行。
   - 提交信息应说明是新增文章、更新索引，还是重构知识库。
   - 推送后记录远端链接和提交哈希。
   - 不强推，不重置，不静默覆盖冲突。

## 更新现有文章的流程

1. 找到文章所在分类和编号。
2. 保持编号不变。
3. 修改正文、图片或隐藏元信息块。
4. 如果分类变化，移动文件，并更新所有索引。
5. 更新 `updated` 日期。
6. 在发布记录或维护清单中记录本次变更。
7. 检查链接、图片和引用。

## 扫描仓库并重建索引

当用户要求“扫描整个仓库”“重新分类”“维护目录”时：

1. 识别正式文章、导航文件、流程文件、素材、脚本、发布包。
2. 只把正式文章纳入文章总索引。
3. 发布包里的 Markdown 不作为正式文章重复入库。
4. 保留原文章编号。
5. 生成或修复总索引、分类索引、系列索引、标签索引。
6. 输出检查摘要：文章总数、编号范围、分类统计、链接检查结果、待处理问题。

## 完成标准

一次知识库维护任务完成时，应能回答：

- 新增或修改了哪些文章？
- 每篇文章的编号、分类、标签是什么？
- 索引是否已更新？
- 图片和本地链接是否可用？
- 发布记录是否已更新？
- 是否已推送到 GitHub？如果已推送，提交哈希是什么？
- 是否有待复查事项？

## 注意事项

- 不要为了分类方便复制同一篇正式文章到多个目录。
- 不要把发布包中的副本误当作正式文章。
- 不要把密钥、Cookie、账号信息或未公开资料写入仓库。
- 涉及最新 AI 工具、模型、价格、新闻、政策和平台规则时，必须核验当前来源并标明截至日期。
- 如果用户想要定期更新热门文章，建议使用 Automation 或 RSS / 搜索 / GitHub MCP，而不是依赖记忆或手动复制。
- 正式文章正文不保留顶部元信息块；文章编号、标题、摘要、分类、标签、状态、封面等统一写入项目内 `07-资料与流程/文章资产登记表.md`。

## 文章质量与微信发布兼容规则

- 正文只保留可发布内容，不展示元信息表、frontmatter 或隐藏元数据块；元信息外置到文章资产登记表、索引和发布记录。
- 正文不重复文章标题，不使用 H1；开头默认使用 `> 一句话结论：……`。
- 封面图外置到素材库和文章资产登记表，不作为正文第一张图重复展示。
- 加粗只用于结论、风险、行动建议和关键限制条件；每篇建议不超过 6 处，超过 10 处视为发布前 P1 问题。
- 禁止 `****`、相邻加粗、整句加粗、单字加粗。
- 微信公众号发布版避免 `md`、`markdown`、`mermaid`、无语言和嵌套代码块；Mermaid 先转 PNG，再作为正文插图引用。
- 结尾使用自然收束或下一步建议，不把“点在看”“关注公众号”等平台 CTA 写入知识库正文。
- 发布或提交前运行项目内文章质量扫描，确认 P0/P1 清零。
