# 文章结构承接与移动端排版检查

用于完整生产包、深度稿、审核修订和去 AI 化改写。目标是避免文章像材料拼接，确保标题、首屏、H2、插图、结尾形成递进。

## 结构地图

写正文前先生成 structure-map.md：

~~~text
标题承诺：
首屏回应：
目标读者：
读者卡点：
核心结论：
H2-1 解决的问题：
H2-2 如何承接 H2-1：
H2-3 如何承接 H2-2：
H2-4 如何承接 H2-3：
第一张图解决什么：
第二张图解决什么：
结尾回收了哪个问题：
读者最终拿走什么：
~~~

## H2 功能分配

每个 H2 只推进一个问题，并承担一个明确功能：

| 类型 | 功能 |
| --- | --- |
| 场景型 | 让读者确认“这说的是我” |
| 解释型 | 把概念讲明白 |
| 方法型 | 给步骤和动作 |
| 例子型 | 降低理解成本 |
| 验收型 | 告诉读者怎么判断结果 |
| 边界型 | 防止误用和过度承诺 |
| 复用型 | 给模板、清单或下一步 |

## 承接检查

- 标题承诺必须在正文前 30% 得到回应。
- 首屏必须包含具体场景、读者问题和一句话结论。
- H2 之间要有递进关系：问题 → 解释 → 方法 → 验收 → 边界 → 复用。
- 每个核心观点都要落到动作、例子、清单、模板或判断标准。
- 每张图前要说明它帮读者看懂什么；图后要自然推进到下一步。
- 结尾必须回收开头问题，并给出低门槛行动。

## 移动端排版检查

- 正文不使用 H1，不重复文章标题。
- 普通段落 1-3 句，一个段落只推进一个意思。
- 连续长段落要拆成小标题、列表、引用或图片。
- 4 列及以上表格默认改成卡片、小节或编号清单。
- 加粗只用于结论、风险、行动建议和关键限制条件；每篇建议不超过 6 处，超过 10 处视为 P1。
- 不使用 Markdown 分割线、中文全角引号、frontmatter、脚注标记、参考资料区块或发布前检查区块。

## 阻断项

- 文章结构跳跃，读者无法理解核心主张。
- 标题、封面、摘要与正文不一致。
- 正文前 30% 未回应标题承诺。
- 结尾只喊口号，没有回收问题或下一步。
