> 一句话结论：把 Codex 用顺，靠的不是长提示词，而是复现、边界、验证三件事。

## 开篇

前两篇讲过入口、配置、权限和复用能力。这一篇只保留实战中最常用的部分：修 bug、提交前安全检查、常用提示词。
遇到问题时，直接套下面的结构发给 Codex。

## 一、修 bug：先复现，再修改

刚开始用 Codex 修 bug 时，很多人会这样写：

> 示例卡：
>
> 这个测试挂了，帮我修一下。

这句话能用，但不够稳。它没有告诉 Codex 是否要先复现、能改哪些文件、修完要跑什么验证。更稳妥的写法是：

> 示例卡：
>
> 请按 Debug 闭环修复这个问题。
>
> 问题：
> [写清楚现象、报错或复现路径]
>
> 上下文：
> - 相关文件可能是：[文件或目录]
> - 相关命令：[测试、构建或启动命令]
>
> 约束：
> - 先复现或写最小失败测试，再修改。
> - 只做最小修复，不做无关重构。
> - 不新增依赖，除非先说明原因。
>
> 完成标准：
> - 说明根因。
> - 列出修改文件和原因。
> - 跑最小相关测试。
> - 如果无法验证，请说明阻塞原因。

这段提示词的关键是边界清楚：问题是什么、从哪里查、不能做什么、怎样算完成。

## 二、排查时盯住三件事

让 Codex 排查问题时，不要只说看一下。直接要求它回答三个问题：

1. 错误在哪里出现。
2. 数据从哪里开始变错。
3. 为什么之前没有被测试覆盖。

前端问题可以这样问：

> 示例卡：
>
> 请沿着数据流排查：UI 参数、请求构造、接口返回、状态更新、页面渲染。确认根因后再修改。

后端问题可以这样问：

> 示例卡：
>
> 请沿着请求链路排查：路由、参数校验、服务层、数据库查询、返回结构。确认根因后再给出最小修复。

如果 Codex 只给出泛泛判断，继续追问：

> 示例卡：
>
> 请指出具体文件、函数和条件分支。不要只说"逻辑有问题"。

## 三、提交前：让 Codex 做一次安全检查

安全检查不用写成很长的制度，提交前发这段就够用：

> 示例卡：
>
> 请在提交前做一次安全检查：
>
> 1. diff 中是否包含密钥、token、Cookie、私钥或真实用户数据。
> 2. 是否新增依赖；如果新增，说明用途和替代方案。
> 3. 是否修改了登录、权限、支付、删除、迁移相关逻辑。
> 4. 是否可能造成数据丢失或接口不兼容。
> 5. 是否运行了最小相关测试、lint、typecheck 或 build。
> 6. 如果有未验证项，请列出原因和建议验证方式。

这里重点记住三条：

- 不要把 API key、数据库密码、私钥、生产 token 直接贴给 Codex。
- 让 Codex 写示例配置时，只写占位符，不写真实值。
- 涉及登录、支付、权限、删除、迁移时，先让 Codex 说明风险，再动手。

如果你怀疑密钥进了文件，先轮换密钥，再清理 Git 记录。不要只把文件里的密钥删掉就结束。

## 四、常用提示词速查

### 解释项目

> 示例卡：
>
> 请先只读分析这个项目，不要修改文件。
>
> 请输出：
> 1. 项目技术栈。
> 2. 主要目录作用。
> 3. 启动、构建、测试命令。
> 4. 新人最该先读的 5 个文件。
> 5. 你发现的风险或不确定点。

### 新增功能

> 示例卡：
>
> 请实现 [功能名称]。
>
> 上下文：
> - 参考现有实现：[文件]
> - UI 或接口要求：[说明]
> - 不要改动：[约束]
>
> 完成标准：
> - 功能可用。
> - 补充必要测试。
> - 跑 lint、typecheck 或相关测试。
> - 最后说明如何验证。

### 审查 diff

> 示例卡：
>
> 请以 code review 方式审查当前 diff。
>
> 优先看：
> 1. 正确性 bug。
> 2. 安全风险。
> 3. 行为回归。
> 4. 缺失测试。
>
> 请给出具体文件和位置，不要只给泛泛建议。

### 最终汇报

> 示例卡：
>
> 请用这个格式结束：
>
> 修改内容：
> - [文件]：[做了什么]
>
> 验证结果：
> - [命令]：[通过、失败或未运行原因]
>
> 风险与后续：
> - [剩余风险或建议]

## 非程序员也能看懂的测试结果

如果你不是开发者，看到测试输出时不用被一长串日志吓到。先看三个信号。

第一，看有没有明确的失败数量。比如“1 failed、12 passed”说明还有问题；如果全是 passed，至少说明这组检查通过了。

第二，看失败位置是否和本次任务相关。如果你只让 Codex 改登录按钮，失败却出现在支付模块，就要让它解释两者是否有关，不要直接接受。

第三，看它有没有说明“修了什么、怎么验证、还剩什么风险”。一个可信的交付，不只说“已完成”，还要能把证据讲清楚。

你可以把这句话作为验收追问：请用非技术语言说明本次失败原因、修改点、验证结果和仍需我人工确认的地方。

## 结尾

Codex 修 bug 要管住过程：先复现，限制改动范围，最后用验证结果收尾。
日常修 bug、提 PR、做提交前检查时，直接复用上面的模板。

## 案例复盘要写清楚失败是怎么被发现的

很多修复类文章只写“我发现问题，然后修好了”，这对读者帮助有限。真正有价值的复盘，必须说明失败信号来自哪里：是测试失败、页面异常、日志报错、用户反馈，还是人工检查发现不一致。信号越具体，后面的定位过程越可信；信号越模糊，修复就越像猜谜。

用 Codex 做修复时，最好把失败现场固定下来：保存报错信息、复现步骤、期望结果和实际结果，再要求它先复述问题，不要立刻改。复述通过后，再让它提出最小修复路径，并说明可能影响哪些地方。最后的验收不能只看“没有报错”，还要看原问题是否复现失败、相关场景是否未被破坏、修改范围是否符合预期。这样一次修复才算可验证，而不是只把红灯变成了绿灯。

## 把一次修复拆成可复现现场

一篇真正有说服力的 Codex 实战，不应该只说“它修好了”，而要让读者看到问题是怎样被定位、怎样被限制范围、怎样被验证的。可以把一次修复写成 5 个镜头。

第一个镜头是失败现场：命令是什么，报错在哪里，复现路径是否稳定。第二个镜头是边界说明：只允许改哪些文件，不允许做哪些顺手重构。第三个镜头是根因判断：是测试断言错了、业务逻辑漏了，还是环境配置不一致。第四个镜头是最小修复：只改让失败通过所必需的部分。第五个镜头是验证结果：同一个失败命令重新运行，相关检查一起通过。

这样写，读者学到的不是某个具体 bug 的答案，而是一套可复用的 Debug 闭环。

## 一条可以直接使用的实战指令

> 示例卡：
>
> 请按可验证修复流程处理这个问题。
>
> 第一步：先复现问题，记录失败命令和关键报错。
> 第二步：判断最可能的根因，并说明还需要查看哪些文件。
> 第三步：只做最小修复，不改无关结构，不新增依赖。
> 第四步：运行最小相关验证，如果失败，继续根据输出修正。
> 第五步：最后汇报根因、修改点、验证命令和剩余风险。

这条指令适合新手，因为它把 Codex 从“直接改代码”拉回到“先证明问题存在”。只要复现和验证还在，AI 编程就不会变成盲改。

## 判断这次修复能不能交付

修复完成后，用 4 个问题验收：失败是否真的复现过；修改是否只围绕根因；验证是否覆盖了原问题；汇报是否能让另一个人接手。

如果 Codex 只给出“我已经修复”，但没有失败输出、修改理由和验证命令，这次任务还不能算完成。可交付的修复，必须留下证据链。

## 为什么要保留过程记录

很多人用 Codex 修完问题后，只保存最后代码，不保存过程。这会带来两个麻烦：第一，下次同类问题还要重新解释一遍；第二，如果修复引入新问题，很难判断是哪一步判断错了。

更好的做法，是让 Codex 在任务结束时留下一个很短的过程记录：复现命令、关键报错、修改文件、验证命令、未处理风险。这个记录不需要很长，但必须能让另一个人接手。

如果你在团队里使用 Codex，这个记录还可以直接变成提交说明或 PR 摘要。它能告诉审查者：这不是一次随手改动，而是一次从失败到验证的闭环修复。

## 新手最适合练的 3 类问题

第一类是测试断言失败。它有明确的失败信号，也容易验证修复是否有效。

第二类是类型或构建错误。它能训练你让 Codex 读取上下文、定位文件、做最小修改。

第三类是文档和代码不一致。它风险较低，但能练习让 Codex 同时理解实现和说明。

不要一开始就让 Codex 重构复杂模块。先从能复现、能回滚、能验证的小问题开始，你会更快建立信任感，也更容易发现它适合做什么、不适合做什么。
