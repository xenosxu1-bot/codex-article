# AI 编程 Agent 实战：非技术人员也能如何把一个想法做成可运行的小工具？

> **核心观点：**你负责判断问题值不值得做，Agent 负责把可验证的步骤往前推。

![显示代码的屏幕，封面文字为后期排版](../08-素材库/图片/12-cover.jpg)

## 非技术人员最容易犯的错：从“做个 App”开始

一个想法要落地，最怕的不是不会写代码，而是没有把想法说成可以验证的任务。你对 Agent 说“做一个帮我管理客户的小工具”，它当然能生成页面；但页面能不能用、数据怎么存、谁能访问、出错怎么办，仍然没有答案。

更好的分工是：**你负责判断问题、场景和取舍；Agent 负责把需求翻译成代码、测试和可运行的版本。**

## 先写一页需求，不要先开代码编辑器

一页就够：

```text
用户：我自己，每周录入 20 条客户跟进
最小目标：新增、查看、搜索、标记“待跟进”
不做：登录、多角色、支付、复杂报表
成功标准：本地能运行；录入一条数据后刷新仍存在；手机浏览不挤
风险：不放真实客户隐私；测试数据必须脱敏
```

这份东西比“帮我做个高级 CRM”更能让 Agent 开工。

## 从想法到运行，按 5 个关口走

| 关口 | Agent 的工作 | 你要验收什么 |
| --- | --- | --- |
| 需求复述 | 复述范围、列出假设 | 有没有把你的问题理解错 |
| 原型 | 给出页面与流程草图 | 顺序是否符合真实使用 |
| 实现 | 建项目、写代码、配置依赖 | 功能是否只做了最小范围 |
| 测试 | 运行检查、列出失败项 | 关键操作是否真的通过 |
| 部署 | 给出部署说明和环境变量清单 | 有没有泄露密钥或测试数据 |

把每一个关口都当成“停一下、看一下”的位置。Agent 可以连续执行，但你不应该连续失去判断。

## 给 Agent 的第一条指令，建议这样写

```text
你是实现协作者。先不要改代码。
请根据需求文档输出：
1. 最小可行版本范围；
2. 页面与数据流；
3. 需要我确认的假设；
4. 实现步骤与测试方式；
5. 安全风险（数据、密钥、权限）。
确认后再按步骤实现，每完成一项报告改动文件和验证结果。
```

这会强迫工作从“生成一堆代码”变成“每一步都能被验证”。

## 不懂代码，也要会看 4 个信号

你不需要先学会读全部源码，但要会问：

- 项目如何启动？复制哪条命令？
- 我要求的 3 个核心操作，分别怎样测试？
- 数据存在哪里？删掉会发生什么？
- 哪些内容绝对不能提交到仓库？

如果 Agent 不能把这些讲成你听得懂的语言，先别急着部署。

## 结尾：把第一版当作提问机器

小工具的第一版不是为了“上线即完美”，而是为了逼你回答：用户会不会点、哪个动作最常用、哪些信息根本没人填。Coding Agent 最有价值的地方，是让你以更低成本获得可运行的反馈；它并不替你决定产品值得不值得继续。Codex 等编程 Agent 的公开文档都强调在任务中检查改动和验证结果，这正是非技术使用者应保留的控制权。[^1]
