> 一句话结论：Dream Skin 可以把 Codex 变成更有个人感的工作台，但它是第三方 CDP 换肤工具；只有"能安装、能做任务、能恢复"三件事都跑通，才值得留下。

我这次没有把"换一张好看的壁纸"当成终点，而是按一次真实使用流程做了验证：从 GitHub 仓库安装、导入矿工猫背景、切换浅色和深色、打开新任务页，再恢复到未加载皮肤的界面。下面所有产品图都来自本机实测；左侧项目、任务名和会话信息已经遮蔽。

## 先判断：你适不适合装

适合：你在 Windows 上高频使用 Microsoft Store 版 Codex Desktop，愿意执行一次验证与恢复，并且能够接受第三方工具在本机打开一个仅回环可访问的调试端点。

不适合：公司受管设备禁止脚本或本地调试端口；你不愿意在更新后重新验证；或者只是想立刻换一张图、并不准备保留恢复入口。

开始前做两件事：

- 保存正在编辑的任务。启动和恢复都可能关闭并重新打开 Codex，未发送的输入可能丢失。
- 只从自己信任的仓库获取代码和背景图；不要把带有窗口、侧栏、文字或按钮的"效果截图"当作背景导入。

## 从仓库安装：先看文件，再复制命令

本文测试的是 Windows + Microsoft Store 版 Codex，Dream Skin 源码要求 Node.js 22 或更高版本。先检查：

> node --version
> Get-AppxPackage -Name OpenAI.Codex

然后克隆项目，进入 `windows` 目录：

> git clone https://github.com/Fei-Away/Codex-Dream-Skin.git
> cd Codex-Dream-Skin\windows

在本次测试所用的源码版本中，Windows 入口脚本位于 `windows` 根目录，因此安装、启动和验证分别是：

> powershell.exe -NoProfile -ExecutionPolicy Bypass -File .\install-dream-skin.ps1
> powershell.exe -NoProfile -ExecutionPolicy Bypass -File .\start-dream-skin.ps1 -PromptRestart
> powershell.exe -NoProfile -ExecutionPolicy Bypass -File .\verify-dream-skin.ps1

这里有一个容易踩坑的地方：仓库说明中部分示例仍写成 `.\scripts\...`，而本次实测源码的入口文件在 `windows` 根目录。**不要机械复制路径；先在当前克隆目录确认 `install-dream-skin.ps1` 实际在哪。**

验证不能只看"界面变了"。本次脚本检查通过时，确认了注入版本与样式存在、装饰层不拦截鼠标、侧栏和输入框仍存在；在新任务首页还确认了主区域、四张建议卡、输入框与无横向溢出。

## 换成自己的背景：矿工猫实测

安装完成后打开系统托盘中的 **Codex Dream Skin - Tray**。建议先保存当前主题，再选择"更换背景图"，导入 PNG、JPEG 或 WebP 的纯背景。项目给出的上限是：文件不超过 16 MB、单边不超过 16384 像素、总像素不超过 5000 万。

这次使用的是用户提供的矿工猫图。它的主体在中下区域，留白足够，放在首页不会压住建议卡和输入框。选图时优先遵守三个原则：

1. 画面不要带任何软件界面或文字；
2. 主体避开输入框、按钮和左侧导航；
3. 把原图、来源和使用授权单独留存，公开文章不要默认把"网上找到的图"当作可再分发素材。

## 浅色、深色都要在"新任务页"试一次

主题里的 `appearance=light / dark` 是 Dream Skin 的显示策略，不等同于 Windows 或 Codex 官方全局主题设置。不要只在首页看一眼：新建一个空任务，检查输入框、项目选择、建议卡、滚动和点击是否正常。

这次实测的结果很明确：

- **浅色**：矿工猫背景在新任务首页完整加载，关键结构检查通过，视觉上也更容易阅读。
- **深色**：皮肤与布局仍能加载，但这张偏白的矿工猫图让底部输入区和部分次级文字对比偏弱。它可以作为效果预览，不建议直接当作长期工作配置；要长期用深色，应换更暗的背景，或重新调色后再测。

## 不喜欢了，如何还原默认 Codex 外观

恢复前先保存工作。最方便的方式是从系统托盘选择"完整恢复 Codex"；也可以回到 `windows` 目录执行：

> powershell.exe -NoProfile -ExecutionPolicy Bypass -File .\restore-dream-skin.ps1 -RestoreBaseTheme -PromptRestart

该操作会关闭正在运行的 Codex、关闭 Dream Skin 的调试会话，再重新打开官方应用。恢复后不要立刻相信"看上去差不多"：新建一个空任务，确认不再出现背景主题卡片，并确认输入框、建议卡和项目切换正常。

## 项目结构与风险：为什么不能只看"好不好看"

以下结论来自对本次测试源码的本地代码阅读和运行验证，不等同于第三方安全审计。

| 层级 | 主要内容 | 使用时的边界 |
| --- | --- | --- |
| Windows 启动层 | `install`、`start`、`restore`、`verify` PowerShell 脚本 | 负责发现 Store 版 Codex、启动和恢复；会重启应用，先保存工作。 |
| 主题与配置层 | 主题保存、图片导入、UTF-8 配置备份与恢复 | 可保存多个主题，但导入图与配置备份都属于本机敏感资产。 |
| 渲染注入层 | `injector.mjs`、CSS 与 renderer 脚本 | 通过 `127.0.0.1` 的 CDP 注入，不修改 WindowsApps 或 `app.asar`；但 CDP 在运行期间对同一台机器上的不可信程序仍是风险面。 |
| 验证与测试层 | `verify` 脚本、Windows 测试脚本 | 能验证结构和注入状态，不能替代人工检查文字对比、隐私和真实任务交互。 |

实测中还遇到过一次"记录的 injector PID 已失效"。正确做法不是看到报错就结束任意 Node 进程，而是先确认状态记录指向的监控进程确实不存在、调试端口属于 Codex，再用项目的恢复或重新应用流程处理。

## 最后的防坑清单

- Codex 更新后皮肤失效：先重新运行安装器，再验证；不要依赖旧的应用路径。
- 端口被占用：不要关闭身份不明的进程，改端口或先确认端口归属。
- 截图准备发布：先隐藏项目、会话、仓库路径、账号和未发送输入；图片中的界面信息也属于隐私。
- 深色不等于可读：背景、卡片、输入框和次级文字都要看对比度。
- 想随时撤回：先确认"恢复官方外观"能跑通，再决定是否常驻皮肤。

一套私人工作台的价值，不是让 Codex 看起来更花，而是在不破坏任务交付、不牺牲隐私、随时能回到原状的前提下，让你愿意更稳定地回到工作里。
