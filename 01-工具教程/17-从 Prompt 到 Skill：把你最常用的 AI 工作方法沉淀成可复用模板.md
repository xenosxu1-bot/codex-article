> **核心观点：**有效提示词不是一段咒语，而是一套带输入、检查和边界的工作方法。

## 为什么好提示词总是“下次失效”？

因为你保存的往往只是一段命令，没有保存它赖以生效的背景：给谁用、输入长什么样、输出要到哪里、哪些错误不能犯、做完怎么检查。

一次有效的提示词，升级成 Skill 后，应该不只是“让 AI 写得更好”，而是让同一类任务**更稳定地被完成。**

## Prompt 和 Skill 的差别

| Prompt | Skill |
| --- | --- |
| 一次输入的指令 | 一类任务的可复用工作流 |
| 关注“这次怎么说” | 关注“每次怎么做” |
| 容易依赖聊天上下文 | 把资料、模板、检查一起保存 |
| 输出好坏靠临场发挥 | 有验收条件和更新记录 |

## 把一个有效方法拆成 5 个零件

以“写一篇公众号文章”为例，别只保存“帮我写文章”。保存这 5 项：

1. **触发条件：**什么任务适合用它；
2. **输入规范：**选题、读者、素材、禁区分别怎么给；
3. **执行步骤：**先 Brief，再资料卡，再初稿，再审校；
4. **输出模板：**Markdown 结构、图片位置、来源格式；
5. **验收清单：**事实、版权、语气、标题、移动端格式。

## 一个最小模板长这样

```text
名称：行业文章生产
适用：需要公开发布、且含时效性事实的长文
输入：选题卡、读者、资料、发布时间、品牌禁区
流程：资料卡 → 结构 → 初稿 → 事实检查 → 排版
完成标准：来源可追溯；图片授权清楚；读者有明确下一步
失败处理：资料不足时只列缺口，不生成确定性结论
```

这已经是一个可复用的 Skill 雏形。

## 什么时候该沉淀？

满足下面任意两条就值得：

- 同类任务一个月出现 3 次以上；
- 你每次都会重复解释同样的背景；
- 团队成员总在相同位置犯错；
- 你已经形成稳定的检查清单；
- 好结果需要多人复用。

Codex 等工具将 Skills 设计为包含说明与资源的可复用目录，正好契合这种“规则 + 模板 + 验收”的沉淀方式。[^1]

## 别把 Skill 写成厚厚的说明书

最常见的失败是把所有经验倒进去，结果谁也不愿意用。先只服务一个交付物；每次出错，就补一条最小规则；每次重复成功，就把输入和检查写清。Skill 的价值不在字数，而在于它能让新人、未来的你、或另一个 Agent 都做对同一件事。

## 结语

Prompt 是一次表达，Skill 是工作方式的存档。你真正要积累的，不是“魔法句子”，而是那些已经被你验证过的路径。
