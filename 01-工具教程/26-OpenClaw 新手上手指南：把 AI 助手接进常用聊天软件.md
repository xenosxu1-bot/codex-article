> 一句话结论：OpenClaw 更像一个个人 AI 助手的本地控制台：你把它跑在自己的设备上，再把 Telegram、Slack、WebChat、移动端节点和各种工具接进来，让 AI 在你熟悉的入口里工作。

如果你只是想找一个网页聊天框，OpenClaw 可能显得重。但如果你想把 AI 助手长期放在自己的电脑、手机和工作流里，让它既能聊天，又能调用工具、记住会话、接入多个消息渠道，它的思路就很值得看。

它的关键词不是又一个聊天机器人，而是个人 AI 助手、Gateway、多渠道、工具和节点。新手先不要被庞大的功能列表吓到，只要按最小闭环跑通一次：安装、引导、打开 Control UI、发出第一条消息，再逐步接入 Telegram 或其他渠道。

## 先用一句话理解 OpenClaw

OpenClaw 官方把它定位为运行在你自己设备上的 personal AI assistant。它可以通过你已经在用的渠道回复你，也可以通过 macOS、iOS、Android 等节点能力扩展到语音、Canvas、设备命令和媒体处理。

先看官方 UI，会更容易理解它和普通聊天机器人的差别。OpenClaw 的界面不是只放一个输入框，而是把 Skills、Commands、Tools 和 Integrations 这些能力放在一个控制台里管理；左侧切换能力类别，中间查看已安装能力，右侧编辑某个 Skill 的说明、允许工具、同步目标和配置项。

![OpenClaw 官方 UI 界面](<../08-素材库/图片/正文插图/26-OpenClaw 新手上手指南：把 AI 助手接进常用聊天软件-官方UI界面.jpg>)

新手不用一开始理解所有配置，先抓住四个基础功能就够了：Skills 决定助手会什么，Commands 像常用快捷指令，Tools 决定它能调用哪些外部能力，Integrations 负责把 OpenClaw 接到其他服务或渠道。右侧的 Sync Targets 可以帮助你理解另一件事：同一份能力配置，可能会同步给不同助手入口，但每个入口能不能执行，仍要看权限和工具边界。

更通俗地说，你可以把它拆成四层：

- 第一层是消息入口：例如 Telegram、Slack、WebChat、iMessage，以及通过插件接入的更多渠道。
- 第二层是 Gateway：负责认证、路由、会话、Control UI 和工具调用入口。
- 第三层是 Agent：接上模型供应商，维护上下文，把任务拆给工具或其他 Agent。
- 第四层是工具与节点：浏览器、执行沙箱、Web 搜索、移动端节点、Canvas、相机、语音等能力。

这也是新手最容易误解的地方。OpenClaw 的重点不是换一个聊天界面，而是把 AI 助手放进一个可扩展的本地控制平面里。

## 新手快速上手：先跑通最小闭环

截至 2026-07-17，官方 Getting Started 给出的新手路径是：准备 Node.js 和模型供应商 API Key，安装 OpenClaw，运行 onboarding，确认 Gateway 正常，再打开 Control UI 发送第一条消息。

![OpenClaw 新手快速上手路径](<../08-素材库/图片/正文插图/26-OpenClaw 新手上手指南：把 AI 助手接进常用聊天软件-正文插图02.png>)

### 第一步：准备环境

你至少需要两样东西。

- Node.js：官方文档建议默认使用 Node 24，并列出 Node 22.22.3+、24.15+ 或 25.9+ 作为可用范围。
- 模型供应商 API Key：可以来自 Anthropic、OpenAI、Google 等供应商，onboarding 会提示你填写。

如果你是 Windows 桌面用户，官方也提供 Windows Hub 作为更容易的入口。对普通新手来说，先按官方推荐路径走，不要一上来就混用 Docker、Nix、WSL、远程 VPS 等方案。

### 第二步：安装 OpenClaw

macOS 或 Linux 的官方安装命令是 `curl -fsSL https://openclaw.ai/install.sh | bash`。

Windows PowerShell 的官方安装命令是 `iwr -useb https://openclaw.ai/install.ps1 | iex`。

如果你已经习惯 npm、pnpm、bun 或 Docker，也可以看官方 Install 章节选择其他安装方式。但第一次体验建议选择最短路径，先证明本机能跑起来。

### 第三步：运行 onboarding

安装完成后运行 `openclaw onboard --install-daemon`。

这个引导会带你选择模型供应商、配置 API Key、设置 Gateway，并按需处理 daemon、渠道、skills 或插件。完整配置可能比几分钟更久，尤其是涉及供应商登录、渠道配对或网络下载时。新手可以先跳过可选项，后面再用 `openclaw configure` 补配置。

### 第四步：确认 Gateway 正常

运行 `openclaw gateway status`。

官方文档说明，默认情况下 Gateway 会监听 18789 端口。如果状态正常，再运行 `openclaw dashboard` 打开浏览器里的 Control UI。

这一步很关键。**先把 Control UI 的单人对话跑通，再去接入更多聊天渠道**。否则你很难判断问题出在模型、Gateway、渠道授权，还是消息路由。

### 第五步：发出第一条消息

在 Control UI 里发一条简单消息，例如让它用三句话解释当前配置。收到回复后，你就完成了最小闭环。

接下来再接一个手机端渠道。官方文档把 Telegram 称为最快的渠道之一，因为只需要 bot token。之后你可以按实际需求扩展 Slack、Discord、Feishu、Signal、WhatsApp、WebChat 等入口。

## 核心功能一：Gateway 是整个系统的控制平面

OpenClaw 的 Gateway 是一个长期运行的进程。官方架构文档把它描述为负责消息面、控制平面和节点连接的核心服务；Control UI、CLI、macOS app、自动化等客户端通过 WebSocket 连接它。

你可以把 Gateway 当成家里的总路由器：消息从不同渠道进来，先到 Gateway，再被分发到 Agent、工具、工作区或节点。默认端口是 18789，默认更偏向本地运行和受控访问。

对新手来说，理解 Gateway 有三个好处：

- 排错更清楚：先看 Gateway 是否运行，再看渠道和模型。
- 权限更清楚：谁能发消息、哪些设备能连接、哪些工具能执行，都不是随便开放。
- 扩展更清楚：以后加渠道、加插件、加节点，都是围绕 Gateway 展开。

## 核心功能二：多渠道入口，而不是只守着一个窗口

OpenClaw 的 README 和 Features 文档列出了大量消息渠道。核心安装包含 iMessage、Telegram、WebChat 等，其他许多渠道通过官方插件或外部插件扩展，例如 Discord、Feishu、Google Chat、LINE、Matrix、Microsoft Teams、Signal、Slack、WhatsApp、Zalo 等。

这让它适合一种很现实的使用方式：你在电脑上用 Control UI 调试，在手机上用 Telegram 发临时任务，在团队工具里只让它处理被提及的消息。

但多渠道不等于全部开放。官方配对文档说明，当 DM policy 设为 pairing 时，未知发送者会收到短码，在你批准前消息不会被处理。配对码会过期，待处理请求也有数量限制。

新手建议很简单：**先接一个私聊渠道，不要一开始接公开群，也不要把 allowlist 放得过宽**。

## 核心功能三：Agent、会话和多 Agent 路由

OpenClaw 不只是把消息转给一个模型。Features 文档提到，它有内置 Agent runtime、tool streaming、多 Agent routing，并且按 workspace 或 sender 做会话隔离。

这意味着你可以把不同任务交给不同 Agent。比如一个 Agent 负责日程和提醒，一个 Agent 负责代码仓库，一个 Agent 负责资料检索。它们共享 Gateway，但会话和权限可以分开设计。

对新手来说，不需要一开始就做复杂多 Agent。你可以先建一个默认助手，再把任务拆成三类：

- 问答类：只需要模型回复，不调用危险工具。
- 整理类：读取资料、总结链接、生成草稿。
- 执行类：涉及文件写入、命令执行、发布、发送、删除，必须加确认。

把任务分清楚，比堆更多工具更重要。

## 核心功能四：工具、Skills、插件和节点

OpenClaw 的工具能力覆盖浏览器自动化、执行、沙箱、Web 搜索、定时任务、Skills、plugins 和 workflow pipelines。它还支持图片、音频、视频、文档等媒体输入输出，以及语音转写、文本转语音和生成类能力。

节点则是另一个重要概念。官方架构文档提到，macOS、iOS、Android 或 headless 节点会以 node 角色连接 Gateway，暴露 Canvas、camera、screen.record、location.get 等命令。节点不是 Gateway，它更像外设和能力扩展。

这部分能力很强，但也最需要克制。新手可以按下面顺序开启：

1. 先启用只读或低风险工具，例如网页检索、资料总结。
2. 再尝试本地文件草稿、项目资料整理等可回滚操作。
3. 最后才考虑执行类工具，并设置人工确认和日志留痕。

## 一个可复现的项目使用案例：开源项目维护助理

下面这个案例不是客户案例，也不是性能承诺，而是一个可以照着搭建的工作流模板。

场景：你维护一个开源项目，平时有 issue、PR、文档更新和发布说明要处理。你希望在手机上给 AI 发消息，让它帮你整理信息、生成草稿，但不让它直接合并、删除或发布。

![OpenClaw 项目助理使用案例](<../08-素材库/图片/正文插图/26-OpenClaw 新手上手指南：把 AI 助手接进常用聊天软件-正文插图03.png>)

### 工作流设计

第一步，入口只开私聊。用 Telegram 或 Control UI 作为项目助理入口，不先接公开群。

第二步，创建一个项目 Agent。给它清晰边界：可以读取指定仓库、整理 issue、生成回复草稿、起草 changelog；禁止直接 push、merge、delete、publish。

第三步，准备三个固定任务。

- 每日 triage：汇总新 issue，按 bug、feature、docs、question 分类，给出优先级建议。
- PR 摘要：读取指定 PR，生成变更摘要、风险点和 reviewer checklist。
- 发布说明草稿：根据已合并 PR 和提交记录，起草 changelog，并标出需要人工核实的条目。

第四步，设置验收标准。每个输出必须带来源链接或文件路径；每个建议必须区分事实、推测和待确认；涉及合并、发版、关闭 issue、删除分支、发外部消息时，必须停止并等待人工确认。

### 一条可以直接发给项目 Agent 的任务说明

你可以这样描述任务：

`请整理最近 24 小时新增 issue，只读取公开 issue 和仓库文件。输出三部分：分类表、需要我今天处理的前三项、可以直接复制但需我确认后发送的回复草稿。不要关闭 issue，不要修改文件，不要调用发布或删除相关操作。`

这条说明的重点不是措辞多漂亮，而是把允许范围、输出格式和禁止动作说清楚。

## 截至 2026-07-17 的使用边界

OpenClaw 的安装方式、节点能力和支持渠道仍可能更新。本文中的命令和路径适合作为新手理解流程的参考，真正操作时应以官方文档和当前版本提示为准。

新手第一次不要追求一次接入所有渠道。更稳的验收顺序是：本机服务能启动，Control UI 能打开，模型能正常回复，一个私聊渠道能收发消息，最后再考虑工具和节点扩展。

如果其中任何一步失败，不要急着继续接 Slack、Telegram 或移动端节点。先把失败步骤、错误提示、系统环境和已执行命令整理出来，再让 AI 或社区帮你定位。多渠道助手最怕的是没跑通基础链路就开始叠功能。

## 新手最容易踩的 5 个坑

### 坑一：把安装成功当成系统可用

安装成功只说明命令可用。真正可用至少要通过 Gateway status、Control UI 打开、模型回复、渠道消息四步验证。

### 坑二：一开始就接太多渠道

渠道越多，排错越复杂。建议先 Control UI，再 Telegram 或 WebChat，最后再接团队渠道。

### 坑三：没有区分只读和执行

整理资料和执行命令不是一个风险等级。能只读就不要写入，能生成草稿就不要直接提交。

### 坑四：忽略配对和 allowlist

多渠道助手一定要先想清楚谁能发消息。对于陌生发送者、群聊和公开入口，默认应该收紧。

### 坑五：把节点当作 Gateway

节点是能力扩展，不是消息中心。手机、Mac、headless node 可以提供能力，但消息和路由仍然要回到 Gateway。

## 适合谁，不适合谁

OpenClaw 适合三类人：

- 想把 AI 助手接进日常聊天软件的人。
- 想在本地或自控设备上运行长期助手的人。
- 愿意花时间配置模型、渠道、工具和安全边界的开发者或高级用户。

它不太适合只想马上打开网页聊天的人，也不适合不愿意维护本地服务、API Key、插件和权限的人。

我的建议是：把 OpenClaw 当成一个可长期扩展的个人 AI 基础设施，而不是一次性玩具。第一次只跑通最小闭环；第二次接入一个私聊渠道；第三次才开始做项目助理、自动化和节点能力。这样你既能看到它的价值，也能把风险控制在自己能理解的范围内。
