> 一句话结论：会提问不够，关键是让 Codex 读懂项目、改得动代码、跑得出验证。

## 开篇

如果你第一次打开 Codex，很容易把它当成更会写代码的聊天机器人。但用起来之后你会发现，它更像一个可以进入项目、读取文件、修改代码、运行命令、查看 diff、继续修复问题的开发协作者。
这篇文章不讲概念堆叠，只解决一个问题：怎样让一个从没用过 Codex 的读者，在 15 分钟内跑通第一个靠谱任务。
这篇会讲清楚：

- Codex 到底适合做什么。
- Codex App、CLI、IDE Extension 分别什么时候用。
- 第一次打开 Codex 应该先问什么。
- 如何写一个让 Codex 不跑偏的任务提示词。
- 如何让 Codex 实现、验证、审查，形成闭环。

## 一、先理解 Codex：它不是只回答问题，而是能执行开发任务

Codex 是 OpenAI 面向软件开发的编程智能体。它能做的事情大致分为五类：

1. 写代码：根据你的目标新增功能、修复逻辑、补齐组件。
2. 读代码：解释项目结构、请求链路、模块职责和风险点。
3. 调试：根据错误日志、失败测试或复现步骤定位根因。
4. 测试：补单元测试、跑 lint、跑 typecheck、跑最小相关测试。
5. 审查：查看当前 diff，指出正确性、回归、安全和缺失测试问题。

不少人用 Codex 效果不稳定，问题常常不在模型，而在于把任务说成了“帮我优化一下”“帮我看看这个项目”。这种提示词太宽，Codex 只能猜。
更稳妥的写法是：把 Codex 当成一个新加入项目的工程师，告诉它目标、上下文、约束和完成标准。

## 二、三个入口怎么选：App、CLI、IDE

### 1. Codex App：最适合新手和可视化工作流

Codex App 是桌面端工作台，适合大多数人入门。
你可以在一个界面里看到：

- 左侧：项目和线程列表。
- 中间：对话、计划、任务过程。
- 右侧：Diff/Review 面板。
- 底部或侧边：终端、Git 操作、文件预览。

适合场景：

- 想边看 Codex 做什么边确认。
- 想查看 diff，再决定是否接受。
- 想用 worktree 隔离实验性改动。
- 想用内置浏览器预览前端页面。

### 2. Codex CLI：适合终端用户和脚本化

CLI 适合已经习惯终端的开发者。进入项目根目录后运行：

> 命令示例：
>
> `codex`

你也可以直接给一个任务：

> 命令示例：
>
> `codex "Explain this codebase to me"`

CLI 的几个高频命令：

> 示例卡：
>
> /plan      先规划，不急着改代码
> /review    审查当前改动
> /model     切换模型和推理强度
> /status    查看会话信息、上下文占用与额度状态
> /diff      查看 Git diff

如果你想把 Codex 接入脚本或 CI 风格流程，可以用：

> 命令示例：
>
> `codex exec "fix the CI failure"`

### 3. IDE Extension：适合边看代码边问

IDE 扩展适合 VS Code、Cursor、Windsurf 等 VS Code 兼容编辑器。
它的优势是能利用编辑器上下文：

- 当前打开的文件。
- 选中的代码片段。
- 你显式 `@` 的文件。

适合这样问：

> 示例卡：
>
> 解释我选中的这段代码在整个请求链路里承担什么职责。

或者：

> 示例卡：
>
> Use @example.tsx as a reference to add a new page named Resources.

如果 Codex 回答像是没看到你正在看的文件，优先检查 IDE context，或者直接把文件 `@` 出来。

## 三、第一次打开 Codex，不要先让它改代码

新手最常见的错误是：刚进项目就说帮我实现某某功能。这会让 Codex 在不了解项目结构、命令、测试方式和代码风格的情况下直接动手。
第一次使用，建议先发这条：

> 示例卡：
>
> 请阅读这个项目的结构，找出：
> 1. 主要源码目录
> 2. 构建、测试、lint 命令
> 3. 项目使用的框架和包管理器
> 4. 重要配置文件
> 5. 你建议写入 AGENTS.md 的项目规则
>
> 先不要修改文件。

这一步的价值很大：你会得到一份项目地图，Codex 也会先收集上下文。

## 四、写一个高质量任务：四件事必须说清楚

一个稳定的 Codex 任务，最好包含四部分：

### 1. Goal：目标是什么

不要说：

> 示例卡：
>
> 优化一下登录。

要说：

> 示例卡：
>
> 修复登录页在密码错误时没有展示后端错误信息的问题。

### 2. Context：上下文在哪里

告诉 Codex 相关文件、错误日志、截图、复现步骤。

> 示例卡：
>
> 相关文件：
> - @src/pages/Login.tsx
> - @src/api/auth.ts
> - @src/pages/Login.test.tsx

### 3. Constraints：哪些不能乱动

例如：

> 示例卡：
>
> 约束：
> - 不改变后端 API shape
> - 不新增依赖
> - 保持现有 UI 文案风格
> - 只做最小修复

### 4. Done when：怎样算完成

例如：

> 示例卡：
>
> 完成标准：
> - 添加或更新回归测试
> - 运行 npm test -- Login.test.tsx
> - 最后汇报根因、改动和验证结果

## 五、可直接复用的新手任务模板

把下面这段复制到 Codex 里，替换尖括号内容即可：

> 示例卡：
>
> 目标：
> <一句话说明要修复或实现什么>
>
> 上下文：
> - 相关文件：@...
> - 当前问题：<错误、日志、截图或用户行为>
> - 类似实现：@...
>
> 约束：
> - 不改变公开 API
> - 不新增依赖，除非先说明必要性
> - 保持现有代码风格
> - 优先做最小改动
>
> 完成标准：
> - 添加或更新测试
> - 运行 <具体命令>
> - 汇报根因、改动摘要和验证结果

## 六、前端任务一定要用浏览器预览

如果你的任务涉及页面、组件、样式或交互，建议让 Codex 打开本地开发服务器，用 in-app browser 检查页面。
可以这样说：

> 示例卡：
>
> 完成后请启动 dev server，打开页面并检查桌面端和移动端布局。
> 如果发现文字溢出、按钮遮挡或布局错位，请继续修复。

前端任务尤其要避免只跑测试不看页面。测试能证明逻辑没坏，但页面是否好用、是否对齐、是否遮挡，还是要看实际渲染结果。

## 七、第一次任务的完整流程

建议你按这个顺序跑：

1. 进入项目根目录。
2. 让 Codex 先阅读项目结构。
3. 让 Codex 总结构建、测试、lint 命令。
4. 选择一个小任务。
5. 用目标、上下文、约束、完成标准写 prompt。
6. 让 Codex 实现。
7. 让 Codex 运行最小相关测试。
8. 用 `/review` 或 App 的 diff 面板审查改动。
9. 你人工确认后再提交。

这套流程看起来多一步，但能显著减少代码写了不少，但不知道对不对的情况。

## 八、新手最容易踩的 5 个坑

### 坑 1：任务太大

“重构整个项目”“优化所有性能问题”都太大。先拆成一个页面、一个模块、一个失败测试。

### 坑 2：不给复现步骤

修 bug 时，复现步骤比一句它坏了重要得多。

### 坑 3：不给完成标准

如果你没说要跑哪个测试，Codex 可能只做静态修改。

### 坑 4：两个线程同时改同一批文件

并行线程很诱人，但同一批文件被两个线程同时修改，容易冲突。

### 坑 5：不看 diff 直接接受

Codex 是强力协作者，不是免审合并按钮。每次都看 diff，尤其是跨模块改动。

## 一个 15 分钟真实任务示例

如果你不知道第一个任务选什么，可以从“修一个看得见的小问题”开始。比如项目里有一个按钮文案不清楚，你可以这样拆：先让 Codex 找到按钮所在文件，再让它只改文案和对应测试，最后运行最小检查。

一个坏任务通常是：帮我优化这个页面。这个说法太大，Codex 不知道优化的是文案、交互、性能还是样式。

一个更好的任务是：找到首页“提交”按钮所在组件，把文案改成“保存设置”；不要改布局；如果有测试就更新测试；完成后告诉我改了哪些文件，并运行最小相关检查。

新手看结果时只看三件事：文件有没有改错、页面文案是否符合目标、检查是否真的跑过。这样做一次，比收藏 20 条提示词更有用。

## 结尾

Codex 的上手关键不是记住所有命令，而是建立正确协作习惯：
先读项目，再说目标。
先给约束，再让它动手。
先跑验证，再看 diff。
最后才接受改动。
当你用这套方式开始，Codex 就不再只是会写代码的聊天框，而是一个能参与真实工程闭环的开发助手。

## 新手最该补上的不是技巧，而是任务边界

第一次使用 Codex 时，很多人会急着让它“帮我改代码”或“帮我做项目”，但真正决定结果质量的，往往是任务边界是否足够清楚。一个高质量任务至少包含五个信息：要改什么、不要改什么、完成后如何检查、哪些文件或页面最重要、遇到不确定时如何处理。只要这五个信息缺失，模型就只能靠猜测推进，结果看起来很努力，却可能偏离你的真实目标。

建议新手从“读懂一个问题并给出修复建议”开始，而不是直接让它大改项目。你可以先要求 Codex 解释当前问题、列出可能影响范围、提出最小修改方案，再决定是否让它执行。这样做会慢一点，但能让你看清它的判断过程，也能及时发现它是否误解了业务规则。等你确认它对项目结构、命名方式、测试入口和交付标准都理解正确，再让它进入修改阶段，成功率会高很多。

## 第一个任务的验收口径

一个适合新手的验收口径可以写得非常朴素：修改范围不超出指定文件；运行或检查步骤能复现；输出里说明改了哪里、为什么改、还剩什么风险；如果没有完成，要说明卡在哪里。这个口径比一句“尽量做好”更有效，因为它把任务从模糊期待变成了可检查的交付物。长期看，你需要积累的不是一堆神奇提示词，而是一组能让任务稳定完成的验收句式。

## 新手第一次任务卡

第一次不要让 Codex 一口气做完整功能，先选一个低风险、能验证、容易回滚的小任务。比如修一个明显报错、补一个缺失测试、改一个文案位置、整理一个 README 小节。

可以直接这样写任务卡：

> 示例卡：
>
> 目标：我想完成一个小改动，而不是重构整个项目。
> 范围：只看和这个问题相关的文件。
> 约束：不要新增依赖，不要改公开接口，不要做无关优化。
> 验证：完成后告诉我改了哪些文件，并运行最小相关检查。
> 汇报：用三句话说明问题、修改、验证结果。

这张任务卡的价值不是让 Codex 更听话，而是让你自己先想清楚：这次任务到底怎么算完成。新手最容易翻车的地方，通常不是不会提问，而是没有定义完成标准。

## 15 分钟验收法

第一次任务完成后，不要只看它说成功了，要按 4 步检查：

1. 看改动范围：是否只动了相关文件。
2. 看运行结果：是否真的跑过测试、构建或最小检查。
3. 看解释质量：是否能说清楚为什么这样改。
4. 看回滚成本：如果改错，能否一眼找到改动并撤回。

只要这四项成立，你就可以把这次任务沉淀成自己的第一个 Codex 使用模板。以后遇到类似问题，不需要重新组织语言，只要替换目标、范围和验证命令即可。
