> 一句话结论：把项目规则写进 Codex 默认上下文。

## 开篇
如果你每次使用 Codex 都要重复说：

- 这个项目用 pnpm，不要用 npm。
- 改 TypeScript 后要跑 typecheck。
- 不要随便新增依赖。
- 不要改公开 API。
- 最后要汇报测试命令。

这些规则不该每次手打，应该沉淀到 `AGENTS.md`。
`AGENTS.md` 是写给 Codex 的项目工作手册。它不替代 README、测试和 lint，只负责告诉 Codex 这个仓库的工作方式、边界和验收标准。

## 一、AGENTS.md 的作用
官方文档把 `AGENTS.md` 归在 Codex 自定义能力里，它适合放持续生效的项目指导，而不是一次性的任务说明。
适合写进去的内容主要有四类：

1. 项目命令：安装、启动、构建、测试、格式化。
2. 代码约定：技术栈、目录边界、命名规则、依赖策略。
3. 完成标准：什么时候可以说任务完成，最后要汇报什么。
4. 代码审查重点：优先检查正确性、安全、回归和缺失测试。

不建议写成泛泛的口号，比如：

```text
- 写出干净代码。
- 遵循最佳实践。
```

更好的写法是：

```text
- 本项目使用 `pnpm`，不要使用 `npm` 或 `yarn`。
- 修改 TypeScript 文件后，至少运行 `pnpm typecheck`。
- 未经明确要求，不要修改公开 API、数据库结构或默认配置。
- 新增运行时依赖前，先说明为什么现有技术栈不够用。
```

规则越具体，Codex 越容易照着执行。

## 二、Codex 会在哪里查找 AGENTS.md
Codex 可以读取多处指导文件，常见位置有两类：

| 位置 | 适合放什么 |
| --- | --- |
| 全局 `AGENTS.md` | 个人沟通偏好、回答风格、常用检查习惯 |
| 仓库内 `AGENTS.md` | 团队规则、项目命令、目录约定、验证标准 |

仓库里也可以有多个 `AGENTS.md`。通常越靠近当前工作目录的文件，越适合写更具体的规则。例如：

```text
my-project/
├─ AGENTS.md                  # 整个仓库通用规则
├─ apps/
│  └─ web/
│     └─ AGENTS.md            # 前端应用规则
└─ packages/
   └─ api/
      └─ AGENTS.md            # 后端或接口包规则
```

如果任务发生在 `apps/web` 下，Codex 会优先参考离当前工作目录更近的规则；根目录的规则负责通用要求，子目录的规则负责局部细节。
这也是官方推荐的思路：把指导写在最接近适用范围的位置，不要把所有规则都塞进根目录一个超长文件。

## 三、配置后 Codex 会怎么执行
配置好 `AGENTS.md` 后，Codex 在处理任务时会出现几个明显变化：

- 开始前会更容易识别项目命令，比如该用 `pnpm test` 还是 `npm test`。
- 修改代码时会更倾向复用现有结构，而不是随手新建抽象。
- 任务收尾时会按规则运行验证命令，或者说明为什么没法运行。
- 做代码审查时会优先关注你写进去的风险点。
- 如果你多次纠正同一个问题，可以让 Codex 把新规则补回 `AGENTS.md`，下一次任务继续继承。

`AGENTS.md` 是指导，不是强制执行器。
真正的防线仍然是测试、lint、typecheck、pre-commit hook 和 CI。`AGENTS.md` 负责说明做法，自动化检查负责证明结果。

## 四、推荐模板：适合大多数项目

![AGENTS.md 模板图](../08-%E7%B4%A0%E6%9D%90%E5%BA%93/%E5%9B%BE%E7%89%87/%E6%AD%A3%E6%96%87%E6%8F%92%E5%9B%BE/04-AGENTS.md%20%E6%B7%B1%E5%BA%A6%E6%A8%A1%E6%9D%BF%EF%BC%9A%E8%AE%A9%20Codex%20%E6%8C%89%E5%9B%A2%E9%98%9F%E8%A7%84%E5%88%99%E5%B7%A5%E4%BD%9C-%E6%AD%A3%E6%96%87%E6%8F%92%E5%9B%BE02.jpg)
可以从这份中文模板开始：

```text
AGENTS.md 示例
## 项目命令

- 安装依赖：`pnpm install`
- 本地开发：`pnpm dev`
- 构建项目：`pnpm build`
- 代码检查：`pnpm lint`
- 类型检查：`pnpm typecheck`
- 运行测试：`pnpm test`

## 编码规则

- 优先复用项目现有模式，不要为了单次任务新增复杂抽象。
- 改动范围要紧扣用户请求，避免顺手重构无关文件。
- 未经明确要求，不要新增运行时依赖。
- 未经明确要求，不要修改公开 API、数据库结构或默认配置。
- 行为变化、bug 修复和边界条件变化，需要补充或更新测试。

## 完成标准

- 任务完成前，运行最小相关验证命令。
- 如果验证无法运行，要说明具体原因和建议用户运行的命令。
- 最终回复需要包含：改了什么、验证结果、未验证项和剩余风险。

## 代码审查重点

- 优先检查正确性、安全风险、行为回归、数据丢失和缺失测试。
- 前端改动要检查响应式布局、文字溢出和交互状态。
- 后端改动要检查参数校验、鉴权、错误处理和迁移安全。
```

## 五、前端项目补充模板
如果项目是 Web、管理后台、小程序或移动端，可以补充：

```text
## 前端项目规则

- 新 UI 优先复用现有组件、样式变量和图标库。
- 不要在页面里堆叠多层卡片，除非现有设计系统就是这样做的。
- 按钮、工具栏、表格、表单控件要保持稳定尺寸，加载和悬停状态不能造成布局跳动。
- 修改可见界面后，尽量同时检查桌面端和移动端展示。
- 文案不能溢出容器，必要时换行或调整布局。
- 如果改动影响页面视觉，尽量启动本地服务并用截图确认。
```

这类规则能减少功能可用但界面粗糙的问题。

## 六、后端项目补充模板
后端项目更关注数据、权限、错误处理和可观测性：

```text
## 后端项目规则

- 外部输入必须在边界层做校验。
- 未经明确要求，不要改变接口返回结构。
- 不要记录密钥、token、密码、Cookie 或用户敏感信息。
- 数据库迁移要尽量保持向后兼容，并说明回滚风险。
- 修改服务逻辑后，需要补充或更新相关测试。
- 涉及登录、权限、支付、删除、迁移的改动，要先说明风险再修改。
```

如果项目涉及生产数据，建议再加一句：

```text
- 任何可能删除、覆盖或批量修改数据的操作，都必须先说明影响范围和验证方式。
```

## 七、单仓多包项目补充模板
单仓多包项目最大的问题是命令容易跑错、包边界容易改乱。可以这样写：

```text
## 单仓多包规则

- 修改前先判断受影响的包或应用。
- 优先运行包级命令，再考虑根目录命令。
- 不要修改共享包，除非任务确实需要。
- 如果共享包有变化，至少检查一个直接使用它的下游项目。
- 不要提交生成文件，除非仓库约定要求提交。
```

这能帮助 Codex 先判断该改哪里，而不是全局搜索后到处动。

## 八、把完成标准写得更具体
很多团队用 Codex 不稳定，原因之一是没有定义完成标准。
建议写进 `AGENTS.md`：

```text
## 完成定义

一个任务只有满足下面条件，才算完成：

- 实现内容符合用户明确提出的范围。
- 已运行最小相关测试、lint、typecheck 或 build。
- 最终回复说明改了什么、验证了什么、哪些没有验证。
- 如果存在高风险假设，要直接说明。
```

这条规则能让 Codex 在最后汇报验证结果，而不是只说已完成。

## 九、哪些内容不适合写进 AGENTS.md
不要把这些内容写进去：

- API key、token、密码。
- 临时任务说明。
- 和项目无关的个人偏好。
- 太长的产品背景文档。
- 会频繁变化的线上数据。

如果是会变化的资料，更适合放在项目文档里，再让 Codex 在任务里读取指定文件。
如果是需要实时访问的系统，比如 Jira、Linear、GitHub Issue、内部文档库，更适合通过 MCP、插件或连接器接入。

## 十、什么时候该更新 AGENTS.md
`AGENTS.md` 不需要一次写完。下面几种情况最适合更新：

- Codex 连续两次犯同一个错误。
- 你在 PR 代码审查里反复提醒同一条规则。
- Codex 找文件时总是绕远路，需要补充目录说明。
- 项目新增了固定验证命令或提交前检查。
- 团队对安全、依赖、数据库迁移有了新的约定。

可以直接让 Codex 帮你补：

```text
请复盘这次为什么改错方向。
把以后能避免这个问题的项目规则整理成 AGENTS.md 条目。
```

## 结尾
提示词解决一次任务，`AGENTS.md` 解决一类任务。
当项目规则、验证命令、代码审查标准和目录边界都写清楚后，Codex 就更像熟悉团队习惯的协作者，而不是每次都从零开始的新同事。
