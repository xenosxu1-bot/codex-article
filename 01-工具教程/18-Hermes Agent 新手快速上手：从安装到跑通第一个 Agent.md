> 一句话结论：Hermes Agent 的新手目标不是先接入一堆工具，而是先完成一个可复查的闭环：安装成功、选好模型、完成一次只读任务、恢复同一段会话。

你可能已经遇到过这种情况：安装命令跑完了，终端里也能看到 `hermes`，但一开始就配置 Gateway、MCP、Skills、Cron，报错立刻变成一团。模型没授权、端点写错、工具权限太大、消息平台 token 缺失，都可能同时出现。

真正让人放弃的，通常不是 Hermes Agent 难，而是第一步把变量加得太多。官方 Quickstart 给出的顺序很明确：先安装，选 Provider，验证一次正常对话；基础对话不稳定前，不要继续叠加 Gateway、Cron、Skills 或路由。本文把这条原则压缩成一套新手可执行路线。

## 先换一个目标：不是安装完成，而是得到一次可验证的交付

很多教程把“命令执行成功”当作完成。对 Agent 来说，这远远不够。你至少要能回答四个问题：

- 它现在在用哪个 Provider 和模型。
- 它能否完成一轮正常对话。
- 它给出的结论能否回到文件或命令核对。
- 重新打开后，它能否接住同一个任务。

这才是第一个 Agent 的最小闭环。它没有炫技，却能把后续排错从“猜配置”变成“定位哪一环出了问题”。

Hermes Agent 的定位也更容易由此看清：它不是单纯的聊天窗口，而是把模型、工具、会话、Skills、消息入口和长期运行能力组合在一起的 Agent 工作台。它适合想把重复流程沉淀下来的人；如果你的任务始终只发生在一个代码仓库、目标是完成一次工程改动，那么把 Codex 当作工程协作者通常更直接。

## 第一步：选对安装路径，不要把系统问题误判成 Agent 问题

当前官方 Quickstart 推荐 macOS 或 Windows 用户优先使用 Hermes Desktop 安装器；只想使用命令行时，再选择系统对应的脚本。Windows 原生 PowerShell 的安装命令是：

```powershell
iex (irm https://hermes-agent.nousresearch.com/install.ps1)
```

Linux、macOS、WSL2 和 Termux 使用：

```bash
curl -fsSL https://hermes-agent.nousresearch.com/install.sh | bash
```

安装结束后，先关闭并重新打开终端，再执行：

```bash
hermes
```

这一步只检查一件事：命令是否真的能启动。不要看到安装日志没有报错，就直接跳到配置机器人或远程服务。Windows 用户尤其不要把 PowerShell、Git Bash、WSL 的环境混在一起；后续配置和运行尽量放在同一套终端环境中。

官方安装方式和平台支持会持续更新，遇到版本差异时，以 [Hermes Agent Quickstart](https://hermes-agent.nousresearch.com/docs/getting-started/quickstart) 和 [Installation](https://hermes-agent.nousresearch.com/docs/getting-started/installation) 为准。

## 第二步：先选模型，再谈 Agent 能力

打开 Hermes 后，先运行：

```bash
hermes model
```

它会引导你选择 Provider 和模型。新手最容易犯的错，是把“装好了 Hermes”和“模型可用”混为一件事。前者只是本地程序存在，后者才意味着认证、模型名、端点和额度都能组成一条可工作的链路。

如果你希望一次配置最少，官方把 `hermes setup` 作为本机快速路径；如果你希望明确控制权限，选择 Blank Slate 的思路更适合练习：先保留最小能力，之后再按需打开工具、Skills、MCP 和消息入口。

这里有一个很实用的判断：**第一天不要为了“功能完整”把所有能力打开，而要为了“错误可定位”让能力尽量少。**

## 第三步：让它完成第一个只读任务

模型配置好后，不要立刻要求它改代码、发消息或操作浏览器。给它一个结果很容易核查的只读任务。进入一个你熟悉、没有敏感信息的项目目录，发送下面这段：

> **第一次任务卡**
> 请先只读当前目录，不要修改任何文件，也不要执行会改变环境的命令。
> 1. 判断这可能是什么项目。
> 2. 找出最可能的启动入口。
> 3. 找出测试、构建或检查命令。
> 4. 每个结论后标明它来自哪个文件；无法确认的内容明确写为推断。
> 5. 最后给我一份不超过 8 条的新手上手清单。

这个任务的价值不在于“让 AI 帮你读目录”，而在于它把验收标准写进了提示词：文件是否存在、命令是否来自 README 或配置文件、推断有没有被伪装成事实，你都能马上检查。

如果第一次回答为空、模型不响应或内容明显不对，不要继续加工具。先回到：

```bash
hermes doctor
hermes model
```

官方文档也把这两个命令放进故障恢复顺序：先诊断配置，再确认 Provider 与模型。

## 第四步：验证会话恢复，才算跑通 Agent 的连续工作

一次对话正常，只说明“这一次调用成功”。Agent 真正有用的地方，是能在后续任务里持续接住上下文。结束第一轮后，用下面的命令恢复：

```bash
hermes --continue
```

然后继续问：

> 基于刚才找到的 README 和配置文件，把上手清单改成一个 30 分钟执行计划。保留文件证据，不确定的地方不要补写。

如果它能利用上一轮已经核对过的材料，而不是从头泛泛介绍，你就完成了第四个验证点。若找不到旧会话，先运行 `hermes sessions list`，确认当前使用的 profile 与会话保存位置；不要急着怀疑记忆功能或安装脚本。

![Hermes Agent 中文30分钟练习卡](<../08-素材库/图片/正文插图/18-Hermes Agent 新手快速上手：从安装到跑通第一个 Agent-正文插图04.png>)

## 30 分钟练习：把第一次成功变成可复用流程

**前 10 分钟：安装与模型。** 启动 `hermes`，完成 `hermes model`，用一句普通问题确认模型会回复。

**第 10 到 20 分钟：只读项目。** 在低风险目录运行第一次任务卡。逐项打开它引用的文件，确认入口、命令和推断边界。

**第 20 到 30 分钟：恢复与复用。** 用 `hermes --continue` 接续任务，让它生成一份 30 分钟上手计划。若失败，按 `hermes doctor → hermes model → hermes sessions list` 的顺序排查。

这套练习完成后，你会得到的不是一句“我已经装上 Agent”，而是一张以后能复用的任务卡和一套自己验证过的排错顺序。

## 为什么 Gateway、Skills 和 MCP 要后置

它们都很有价值，但每增加一层，问题空间就会扩大。

- **Gateway** 会增加 bot token、消息平台、身份白名单和触发规则。
- **Skills** 会增加第三方内容、执行步骤和安全审查。
- **MCP 与外部工具** 会增加网络、账号、读写权限与数据边界。

所以更稳的顺序是：先本机 CLI，再低风险只读任务；然后恢复会话；之后才根据真实需求启用一个工具或一个 Skill。需要连接 Telegram、Discord、Slack 等入口时，再运行 `hermes gateway setup`。每加一层，都重新做一次小范围验收。

尤其是涉及文件写入、命令执行、联网搜索或账号访问时，不要把权限交给“看起来聪明”的提示词。先限定目录、操作类型和批准方式，再观察它是否按边界执行。

## 把第一天的目标收窄成这四件事

今天先完成：

1. 在一种明确的终端环境里启动 `hermes`。
2. 通过 `hermes model` 配好一个能稳定回复的模型。
3. 用第一次任务卡完成一个可核查的只读任务。
4. 用 `hermes --continue` 接续同一个任务。

四件事都通过后，你再决定是否要增加 Skills、工具、Gateway 或 MCP。这样做看起来慢一点，实际上会省掉大部分“装好了但不知道哪里坏了”的时间。

下一次打开 Hermes，不必从复杂的自动化开始。找一个你熟悉的目录，跑完第一次任务卡，把每条结论都对回文件；这就是最值得拥有的第一个 Agent。
