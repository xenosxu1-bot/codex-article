第一次装 Hermes，最容易浪费时间的不是安装命令，而是把模型、工具、消息平台和定时任务一起打开。任何一处配置没对，报错都会混在一起，最后连第一步出了什么问题都说不清。

这篇只解决新手第一天最该做的事：在 30 分钟内跑通一个可核查的闭环。你需要看到 `hermes` 能启动、模型能回复、它能完成一次低风险的目录阅读，并且能接着同一段会话继续工作。

## 先定验收标准，别只看安装日志

终端里出现 `hermes`，只说明程序已经装进系统。第一轮上手完成前，至少核对下面四件事：

- 当前使用的是哪个 Provider 和模型。
- 模型能否完成一轮正常对话。
- 它给出的项目结论能否回到文件或命令核对。
- 重新打开后，能否接着同一个任务往下做。

这四项都能通过，后面的排错才有起点。否则，即使接上了 Gateway、Skills 或 MCP，问题也只会更难定位。

## 安装前先把运行环境定死

官方 Quickstart 建议 macOS 和 Windows 用户优先考虑 Hermes Desktop。只使用命令行时，再按当前系统选择安装脚本。

Windows 原生 PowerShell：

```powershell
iex (irm https://hermes-agent.nousresearch.com/install.ps1)
```

Linux、macOS、WSL2 和 Termux：

```bash
curl -fsSL https://hermes-agent.nousresearch.com/install.sh | bash
```

除 Windows 外，安装器唯一前置是 Git；Linux 还需要 `curl` 和 `xz-utils`。安装后重新打开终端，再运行：

```bash
hermes
```

这一步只确认命令能否真正启动。Windows 用户尤其要避免在 PowerShell、Git Bash 与 WSL 之间来回切换：后续配置和运行尽量放在同一套终端环境里。安装方式与平台支持会更新，遇到版本差异时，以 [Hermes Agent Quickstart](https://hermes-agent.nousresearch.com/docs/getting-started/quickstart) 和 [Installation](https://hermes-agent.nousresearch.com/docs/getting-started/installation) 为准。

## 先把模型跑通，再增加 Agent 能力

启动 Hermes 后，先运行：

```bash
hermes model
```

它会引导你选择 Provider 和模型。这里检查的不是模型名称好不好看，而是认证、端点、模型权限和额度能否一起工作。官方 Quickstart 还建议为 Agent 选择至少 64K 上下文窗口的模型；上下文太短时，任务一长就容易丢掉前面的文件证据和限制条件。

新安装后运行 `hermes setup` 会给出三种模式。Quick Setup 是官方推荐的快速路径；若你更在意每一项能力的边界，可以从 Blank Slate 开始。它不是零权限模式，默认仍保留 File Operations 和 Terminal 两组工具。先用 `hermes tools` 看清当前工具范围，再决定是否扩大能力。

配置完成后，先问一个不需要调用工具的普通问题，再记录本次使用的 Provider、模型标识和 profile。后面报错时，这三项比一张只截到错误提示的图片更有用。不要把 API Key、完整配置文件或终端历史直接贴进对话；排错时只提供必要的错误片段。

第一天的原则很简单：少开一项能力，就少一种排错变量。

## 第一个任务选只读目录，但别把提示词当权限控制

模型配置好后，先不要让它改代码、发消息或操作浏览器。选一个没有密钥、没有客户数据的测试目录，最好是项目副本，然后发出下面这张任务卡：

> **第一次任务卡**
>
> 请先只读当前目录，不要修改任何文件，也不要执行会改变环境的命令。
>
> 1. 判断这可能是什么项目。
> 2. 找出最可能的启动入口。
> 3. 找出测试、构建或检查命令。
> 4. 每个结论后标明它来自哪个文件；无法确认的内容明确写为推断。
> 5. 最后给我一份不超过 8 条的新手上手清单。

这张任务卡的作用是把验收标准写清楚：文件是否存在、命令是否来自 README 或配置文件、推断有没有被写成事实，都能马上检查。

第一次输出不必追求完整方案。先检查它是否列出了项目入口、依据文件和一个可运行的验证命令；这三项对得上，才值得继续让它整理计划或提出改动。

但它只是对模型的工作要求，不会自动收回 Hermes 已获得的文件或终端权限。要做严格隔离，应从工具配置和运行目录下手；第一次练习不要放在生产项目、真实账号目录或含敏感文件的工作区。

如果模型不响应、回答为空，或结论明显对不上文件，先停在基础链路排查：

```bash
hermes doctor
hermes model
hermes setup
```

先看诊断结果，再确认 Provider 和模型；只有基础对话稳定后，才值得继续加工具。

## 会话能接上，才有连续工作的价值

第一轮任务结束后，运行：

```bash
hermes --continue
```

接着发送：

> 基于刚才找到的 README 和配置文件，把上手清单改成一个 30 分钟执行计划。保留文件证据，不确定的地方不要补写。

这一步的验收不是看它写得多流畅，而是看它会不会沿用前一轮已经核对过的文件。找不到旧会话时，先运行：

```bash
hermes sessions list
```

确认自己仍在同一个 profile 与会话范围内，再决定是否重开任务。不要一看到上下文断掉，就先怀疑安装脚本或记忆功能。

## 30 分钟内怎么跑完第一轮

**前 5 分钟：固定环境。** 选定一种终端，安装后启动 `hermes`。命令不能启动，就只处理环境问题。

**第 5 到 10 分钟：配置模型。** 运行 `hermes model`，发一个普通问题，确认模型确实能返回内容。

**第 10 到 20 分钟：做一次目录阅读。** 在测试目录执行第一次任务卡。逐项核对它引用的文件、入口和命令。

**第 20 到 30 分钟：续接会话。** 使用 `hermes --continue`，让它基于已有证据整理计划。失败时按 `hermes doctor → hermes model → hermes setup → hermes sessions list` 的顺序缩小范围。

完成后，保留两样东西：那张任务卡，以及本次实际遇到的诊断结果。下次换项目或换 Provider 时，它们比一份泛泛的教程更有用。

## Gateway、Skills 和 MCP 放到第二轮

这些能力不是不能用，而是不该在基础链路还没验证时一起开启。

- **Gateway** 可能引入 bot token、消息平台、身份白名单和触发规则。
- **Skills** 需要核对安装来源、执行步骤与外部依赖。
- **MCP 和外部工具** 往往涉及网络、账号、读写权限和数据边界。

需要连接 Telegram、Discord、Slack 等入口时，再运行 `hermes gateway setup`。每新增一个连接，只做一个小任务，并重新检查它引用的文件、权限和输出；不要把联网、写文件、执行命令和账号访问同时交给一个刚配好的 Agent。

今天的完成条件只有四项：`hermes` 能启动、模型能回复、只读任务的结论能复查、`hermes --continue` 能续接同一任务。完成后再根据真实需求选择下一步：要做消息机器人就研究 Gateway，要接外部服务就研究 MCP，要沉淀流程再考虑 Skills。一次只加一层，问题才知道该从哪里查。
